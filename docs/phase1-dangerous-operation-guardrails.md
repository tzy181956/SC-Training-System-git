# 第一阶段危险操作保护

## 1. 这一步只解决什么问题

本文件只收口第一阶段的危险操作保护，目标是：

1. 危险操作不再像普通按钮一样直接执行
2. 执行前必须看到影响范围
3. 高危操作必须输入确认词
4. 执行后必须写危险操作日志

本步不展开：

- 复杂恢复向导
- 日志页复杂筛选 UI
- 批准流和多人审批

---

## 2. 当前纳入保护的操作

### 前端管理页

- 删除训练模板
- 删除模板动作
- 删除动作
- 删除训练记录（课后组记录）
- 删除当前筛选批次的测试数据

### 后端 / 脚本

- EXOS 动作库覆盖导入
- 真实数据清空后重导脚本
- 备份恢复脚本

---

## 3. 二次确认规则

### 普通危险操作

执行前必须明确确认：

- 删除训练模板
- 删除模板动作
- 删除动作
- 删除训练记录
- 删除测试数据批次
- EXOS 覆盖导入

确认框必须至少说明：

1. 操作对象
2. 影响范围
3. 建议先确认备份

### 高危操作

以下操作除了确认，还必须输入确认词：

- 清空后重导真实数据  
  确认词：`DELETE REAL DATA`
- 恢复备份覆盖当前数据库  
  确认词：`RESTORE_BACKUP`

---

## 4. 日志落点

危险操作日志统一落到：

```text
dangerous_operation_logs
```

当前最小字段包括：

- `operation_key`
- `object_type`
- `object_id`
- `actor_name`
- `source`
- `status`
- `summary`
- `impact_scope`
- `confirmation_required`
- `confirmation_phrase`
- `backup_path`
- `extra_data`
- `created_at`

---

## 5. 备份联动

危险操作在第一阶段默认和备份机制联动：

- 删除类和覆盖类操作先触发危险操作备份
- 恢复备份前，会先给当前线上库再做一份兜底备份

---

## 6. 当前边界

这一步仍未做：

- 恢复前差异预览
- 复杂审批
- 更细粒度的对象权限
- 按单个对象的精细恢复授权
