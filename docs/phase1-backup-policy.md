# 第一阶段备份触发机制

## 1. 这一步只解决什么问题

本文件只定义第一阶段的备份触发机制，目标是把“备份”变成默认动作，而不是出事后补做。

本步只覆盖：

1. 每日自动备份
2. 迁移前自动备份
3. 危险操作前自动备份
4. 最近几天 + 最近几周的保留策略

本步不展开：

- 复杂备份管理界面
- 远程备份和多副本存储
- 复杂差异预览和高级增量恢复

---

## 2. 备份文件位置

当前 SQLite 主库默认位于：

```text
backend/training.db
```

统一备份目录位于：

```text
backend/backups/
```

该目录已加入 `.gitignore`，不会进入仓库版本管理。

---

## 3. 命名规则

受系统管理的备份文件统一命名为：

```text
training-YYYYMMDD-HHMMSS-trigger-label.db
```

实际格式来自：

```text
{数据库文件名}-{时间戳}-{触发来源}[-标签].db
```

当前主库文件名为 `training.db`，所以默认前缀是 `training-`。

示例：

```text
training-20260424-093000-daily_auto-scheduled.db
training-20260424-101500-before_migration-upgrade_from_c318e7988c37.db
training-20260424-103000-before_dangerous-delete_template_12.db
```

---

## 4. 触发时机

### 4.1 每日自动备份

- 后端启动时先检查当天是否已有 `daily_auto` 备份
- 如果当天还没有，则立即创建一份
- 后端常驻后，每小时轮询一次
- 同一天只保留一份自动日备份，不重复生成

### 4.2 迁移前自动备份

迁移脚本在以下入口先备份，再继续：

- `python backend/scripts/migrate_db.py bootstrap`
- `python backend/scripts/migrate_db.py ensure`

触发标记为：

```text
before_migration
```

### 4.3 危险操作前自动备份

当前已接入的第一批危险操作：

- 删除模板动作
- 删除训练模板
- 删除动作
- EXOS 动作库 `replace_existing=True` 导入
- 测试数据清空后重导脚本

触发标记为：

```text
before_dangerous
```

---

## 5. 保留策略

第一版保留策略固定为：

1. 最近 `7` 天：保留所有系统管理备份
2. 超过最近 7 天、但仍在最近 `8` 周内：每周只保留最新一份
3. 超过最近 8 周：由系统自动清理

说明：

- 只清理符合系统命名规则的“受管理备份”
- 手工命名或外部复制的 `.db` 文件不会被这套策略误删

---

## 6. 实现边界

当前实现使用 SQLite 原生 `backup()` API 复制数据库，不依赖简单文件拷贝，目的是降低数据库正在使用时的备份风险。

当前只支持：

- SQLite 数据库

当前已支持：

- 整库恢复
- 训练数据恢复
- 测试数据恢复

当前不处理：

- 备份压缩
- 异地备份
- 单运动员 / 单堂课 / 单模板的细粒度恢复
- 恢复前差异预览向导

---

## 7. 验收关注点

人工验收至少看这几件事：

1. 执行危险操作前，`backend/backups/` 下出现新备份
2. 执行迁移前，先出现新备份，再进入迁移
3. 后端启动后，当天首次运行会生成 `daily_auto` 备份
4. 备份文件名能直接看出时间、触发来源和标签
5. 备份目录会按“最近几天 + 最近几周”自动收敛
