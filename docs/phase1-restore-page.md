# 第一阶段重构补充｜备份恢复页基础版

## 1. 这一步只解决什么

本文件只记录第一阶段中的“备份恢复页基础版”，目标是：

1. 管理端能从备份列表里直接发起恢复。
2. 第一版先支持：
   - 整库恢复
   - 某类数据恢复
3. 恢复前必须二次确认、明确影响范围、建议先备份，并要求输入确认词。
4. 恢复后必须明确提示：系统恢复到了哪个时间点。
5. 恢复操作必须写日志，且日志可被日志页查询到。

本步不展开：

- 复杂差异预览
- 单个运动员 / 单堂课 / 单张表的细粒度恢复向导
- 高级增量恢复

---

## 2. 页面与入口

当前恢复页落点：

- 页面：`frontend/src/views/BackupsView.vue`
- 路由：`/backups`
- 导航入口：`frontend/src/components/layout/AppShell.vue`
- 管理首页快捷入口：`frontend/src/views/DashboardView.vue`

页面结构保持第一阶段的“实用优先”：

1. 顶部显示备份目录、命名规则、保留策略。
2. 左侧显示备份列表。
3. 右侧显示所选备份详情、恢复范围、影响范围说明、恢复按钮。

---

## 3. 后端接口

本步新增后端接口：

- `GET /api/backups`
  - 返回备份列表
  - 返回备份目录、命名规则、保留策略
  - 返回当前支持的恢复范围
- `POST /api/backups/restore`
  - 执行恢复
  - 要求 `confirmed = true`
  - 要求确认词 `RESTORE_BACKUP`

核心文件：

- `backend/app/api/endpoints/backups.py`
- `backend/app/schemas/backup_restore.py`
- `backend/app/services/backup_service.py`

---

## 4. 当前支持的恢复范围

### 4.1 整库恢复

恢复范围：

- 当前正式 SQLite 数据库整体覆盖

特点：

- 恢复力度最大
- 最适合“当前线上数据整体错乱”或“需要完整回到某个时间点”的情况
- 恢复后会按当前运行时 schema 再做一次兼容补齐

### 4.2 训练数据恢复

第一版只恢复以下训练执行相关表：

- `training_sessions`
- `training_session_items`
- `set_records`
- `training_session_edit_logs`
- `training_sync_conflicts`
- `training_sync_issues`

边界：

- 不恢复模板
- 不恢复动作库
- 不恢复测试数据

为了避免恢复出“断引用”的训练记录，本步在执行前会校验：

- 运动员是否仍存在
- 计划分配是否仍存在
- 模板是否仍存在
- 模板动作项是否仍存在
- 动作是否仍存在

如果这些基础依赖已经不在当前库中，就拒绝部分恢复，并提示改用整库恢复。

### 4.3 测试数据恢复

第一版只恢复：

- `test_records`

执行前会校验备份里引用的运动员是否仍存在。  
若当前库缺少这些运动员，就拒绝部分恢复，避免导入出悬空测试记录。

---

## 5. 恢复前保护

本步复用第一阶段危险操作规则：

1. 恢复前先弹二次确认
2. 明确影响范围
3. 建议先备份
4. 强制输入确认词 `RESTORE_BACKUP`

除此之外，系统还会自动做一件事：

- 在真正执行恢复前，先对**当前线上数据库**做一份兜底备份

也就是说，恢复动作本身依然满足“先备份，再覆盖”。

---

## 6. 恢复执行规则

### 6.1 备份来源

当前恢复页只允许从 `backend/backups/` 目录中的备份列表恢复，避免任意路径覆盖。

备份列表支持两类文件：

1. 当前受系统管理的标准命名备份  
   例如：`training-20260424-220435-daily_auto-scheduled.db`
2. 旧版迁移备份  
   例如：`training.migration-backup-20260424-092045.db`

### 6.2 整库恢复执行方式

整库恢复不是直接把备份文件硬拷贝到线上文件，而是：

1. 先用 SQLite `backup()` API 复制到临时文件
2. 再原子替换当前数据库文件
3. 最后执行运行时 schema 补齐

这样做的目的，是尽量降低恢复中间态损坏数据库文件的风险。

### 6.3 某类数据恢复执行方式

第一版某类数据恢复采用：

1. 附加备份数据库
2. 校验关键外键依赖
3. 在事务内删除当前范围数据
4. 按表从备份导回

这一步仍然是“覆盖式恢复”，不是合并算法。

---

## 7. 恢复结果提示

恢复成功后，页面会明确提示：

1. 恢复使用的是哪个备份文件
2. 恢复到了哪个时间点
3. 恢复前自动生成的兜底备份文件路径

这一步是为了满足第一阶段要求：

> 恢复后必须明确提示恢复到了哪个时间点

---

## 8. 日志落点

恢复动作日志继续落到：

```text
dangerous_operation_logs
```

本步新增的日志语义包括：

- `operation_key = restore_backup`
- `object_type = database / training_records / test_records`
- `impact_scope` 中记录：
  - 原备份文件
  - 恢复范围
  - 恢复时间点
  - 涉及表
- `backup_path` 记录恢复前兜底备份路径

日志页通过现有统一接口即可看到这些记录：

- `GET /api/logs`

---

## 9. 人工验收

本步建议至少验这几件事：

1. 打开 `/backups` 能看到备份列表
2. 选中一个备份后，右侧能看到恢复范围与影响说明
3. 点击恢复时，必须先确认并输入 `RESTORE_BACKUP`
4. 恢复成功后，页面能明确看到“恢复到了哪个时间点”
5. 恢复后，日志页能查到本次恢复操作

---

## 10. 当前边界

这一步做完后，仍然没有展开：

- 恢复前数据差异预览
- 单运动员恢复
- 单堂训练课恢复
- 单模板 / 单动作库条目的细粒度恢复向导
- 复杂运维统计图

第一版先把“能从备份列表恢复、能确认、能记录、能提示时间点”这条最小闭环做稳。
