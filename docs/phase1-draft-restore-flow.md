# 第一阶段：训练端草稿恢复与继续录课

> 文档状态：当前有效专题  
> 当前用途：说明训练端本地草稿恢复与继续录课链路。  
> 注意：本专题只覆盖当前浏览器/当前设备内的草稿恢复，不代表跨设备强持久化方案。  

## 本步范围

本步只处理训练端“设备重开后的恢复提示与继续录课”：

- 检测本机是否存在可恢复的本地草稿
- 给出“继续 / 放弃”两条明确路径
- 继续后优先回到动作列表
- 不重排训练端整体界面

本步不处理：

- 新的同步算法
- 复杂冲突界面
- 训练端 UI 大改

当前恢复链路基于本机当前浏览器里的 `localStorage` 草稿，主要覆盖同一浏览器 / 同一设备内的恢复；不等同于跨设备、跨浏览器或清缓存后仍可恢复的强持久化方案。

## 恢复入口

### 1. 训练模式入口页

`TrainingModeView` 打开时扫描本地草稿列表。只要存在满足以下任一条件的草稿，就弹出恢复提示：

- 训练课未完成
- 或者虽然课已结束，但仍有未同步数据

恢复提示展示草稿的运动员、计划、日期、同步状态和最后保存时间。

### 2. 训练会话页

如果用户直接打开某堂训练课，或从恢复提示点击“继续录课”进入训练页，`TrainingSessionView` 会再次按当前课上下文检查草稿：

- 找到可恢复草稿：进入恢复流程
- 找不到可恢复草稿：继续使用当前后端记录

## 两条恢复路径

### 继续

点击“继续录课”后：

1. 保留当前草稿的 `session_key`
2. 进入原训练课对应的训练页
3. 恢复本地草稿快照
4. 优先停留在动作列表
5. 保留上次动作的高亮定位
6. 通过轻提示告知“请从动作列表继续录课”

这里故意不自动打开录组面板，避免设备重开后直接把用户丢进某个动作详情，造成迷路。

### 放弃

点击“放弃草稿”后：

1. 直接删除这份本地草稿
2. 不再弹出这份草稿的恢复提示
3. 不创建新的 `session`
4. 用户继续按当前页面正常选择训练计划

## 不重复创建 Session 的规则

恢复链路仍遵守第一阶段主规则：

- 打开计划只返回预览或既有训练课
- 第一组提交成功后才正式创建 / 确认 `session`
- 恢复草稿只恢复本地快照，不额外新增建课路径

如果恢复过程中后端已经存在正式 `session`，前端会在路由切换时保留恢复上下文，再进入该 `session` 页面恢复草稿，避免因为路由变化丢掉恢复流程。

## 验收点

### 人工验收

1. 在训练页录入至少一组，关闭页面或刷新浏览器
2. 重新进入训练模式页，确认出现恢复提示
3. 点击“继续录课”，确认进入原训练课并优先停留在动作列表
4. 确认上次进行中的动作仍保持高亮
5. 点击高亮动作后，确认可以继续录组
6. 再次制造草稿后点击“放弃草稿”，确认本地草稿被删除且不再重复提示

### 技术自测

- 恢复流程不新增重复 `session`
- 预览路由切换为正式 `session` 路由后，恢复上下文不丢失
- 本地草稿恢复后，录组仍沿用现有本地优先保存和后台同步链路
