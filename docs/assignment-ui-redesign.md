# Assignment UI Redesign

> 文档状态：当前有效专题  
> 当前用途：说明“计划分配”页面的前端视觉与交互改版方案。  
> 目标阶段：管理端第二轮体验收口。  
> 注意：本方案优先聚焦前端 UI/UX，后端只建议做少量接口补充，不改核心分配业务规则。  

## 1. 本步范围

### 本步只解决

- 计划分配页面视觉层级混乱
- “查看现状”和“新增分配”混在一个页面导致的认知负担
- 教练按队批量分配效率低
- 右侧预览过重、主流程被打断
- 日期、筛选、分配步骤之间的界面语义不清

### 本步不解决

- 训练模板系统重写
- 分配业务规则重写
- 多维高级排课
- 大规模历史分配编辑
- 复杂权限系统

### 影响模块

- `frontend/src/views/PlanAssignmentsView.vue`
- `frontend/src/components/assignment/AssignmentPreviewPanel.vue`
- 可能新增的 `frontend/src/components/assignment/*`
- 如需少量后端补充：`backend/app/schemas/assignment.py`、`backend/app/services/assignment_service.py`

### 数据风险

- 纯前端改版本身无数据风险
- 若补充后端字段，建议只新增只读返回字段，不改表结构，不改分配写库逻辑

---

## 2. 现状问题总结

当前页面的主要问题不是“样式不好看”，而是信息架构与教练工作流不一致。

### 2.1 一个页面承担了太多任务

当前页面同时承担：

- 看当前和后续已有分配
- 看未分配队员
- 批量选人
- 选模板
- 设日期
- 设循环星期
- 看分配预览
- 批量取消分配

这会导致教练在“查看”与“操作”之间频繁切换，视线来回跳。

### 2.2 顶部空态占空间，但没有推进下一步

空态下，“当前与后续计划”大卡片高度很高，但实际只传达一句“还没有有效计划组”。  
它占了首屏大量空间，却没有让教练更快进入分配动作。

### 2.3 同一批人被重复展示

页面同时存在：

- 右侧“未分配队员”名单
- 下方“选择队员”卡片区

本质上在重复呈现同一批对象，信息密度高，但操作收益低。

### 2.4 步骤顺序不符合教练分配习惯

真实场景里，教练更常见的思路是：

1. 先决定给哪个队 / 哪类人分
2. 再决定分哪个模板
3. 再决定什么日期段执行
4. 最后确认人员与预览

当前页面却是先铺开全部运动员，再选模板和时间，导致前两屏都是“找人”。

### 2.5 右侧预览过重

当前右侧预览同时展示：

- 已选人数
- 日期与循环星期
- 模板模块
- 动作明细
- 异常提示

这会压缩主操作区宽度，也让“第二步选模板”和“第三步设时间”被挤到更下面。

### 2.6 关键状态被共用，造成语义混乱

当前页面里，概览区“查看日期”和新建分配里的“开始日期”共用 `form.start_date`。  
这意味着用户调新建分配时间时，也会同步改变上方概览口径。

相关位置：

- `frontend/src/views/PlanAssignmentsView.vue`
- `form.start_date`
- `loadOverview(form.start_date)`
- 概览区日期输入
- 新建分配区开始日期输入

这会带来明显的产品语义混乱。

---

## 3. 改版目标

改版后的计划分配页要优先满足下面四个目标：

1. 教练在首屏 3 秒内看懂“现在看的是现状还是在做新分配”
2. 教练能按队伍或范围快速批量分配，而不是逐个卡片点人
3. 页面主流程从上到下连续，不依赖频繁滚动和来回对照
4. 分配确认前能看清风险和摘要，但不会被动作明细淹没

---

## 4. 推荐页面结构

推荐把当前单页拆成两个主视图，但仍保留在同一路由下：

- `现有分配`
- `新建分配`

不需要拆成两个新路由，先做页面内切换即可。

### 4.1 页面顶层结构

推荐顶层结构：

1. 页面标题区
2. 顶部筛选与视图切换栏
3. 主内容区
4. 右侧摘要 / 详情面板

推荐示意：

```text
计划分配
---------------------------------------------------------
[查看日期] [项目] [队伍]   [现有分配] [新建分配]
---------------------------------------------------------

现有分配视图：
左：计划组列表 / 空态
右：未分配摘要 / 计划组详情 / 取消分配操作

新建分配视图：
左：步骤流（范围 -> 模板 -> 时间 -> 确认对象）
右：sticky 分配摘要与风险提示
```

## 5. 现有分配视图方案

“现有分配”视图负责看现状和做少量取消，不负责新建主流程。

### 5.1 视觉目标

- 压缩空态高度
- 强化“今天生效 / 即将开始”区别
- 减少嵌套滚动
- 取消分配入口不抢主视觉

### 5.2 建议布局

左侧主区：

- 顶部 3 个横向摘要卡
- 下方计划组列表
- 空态时直接给出“去新建分配”按钮

右侧副区：

- 未分配人数摘要
- 队伍分布摘要
- 选中计划组后的详情区

### 5.3 摘要卡改法

保留三张卡，但改成更紧凑的条状卡：

- 已安排队员数
- 今日有效计划组
- 未分配队员数

不建议继续用大面积留白卡片。

### 5.4 计划组卡片改法

每个计划组卡只保留最关键的信息：

- 模板名
- 时间段
- 执行日
- 当前状态：`进行中` / `即将开始`
- 人数
- 已覆盖队伍标签
- 展开查看队员名单

当前的队员 chip 可改成：

- 默认只显示前 4 到 6 人
- 超出显示 `+N 人`
- 点击卡片再展开完整名单

这样能显著压缩列表高度。

### 5.5 取消分配交互改法

不建议在每个卡片内部直接大面积显示“确认取消”流程。  
建议改成：

1. 选中计划组
2. 右侧详情区显示该组完整名单
3. 在右侧勾选要取消的人
4. 点击“取消所选分配”
5. 再做二次确认

这样危险动作和浏览动作边界更清楚。

## 6. 新建分配视图方案

“新建分配”是页面主流程，应该成为默认主入口。

### 6.1 核心改法

把当前顺序：

- 选人
- 选模板
- 选时间
- 右侧预览

改成：

1. 选择分配范围
2. 选择训练模板
3. 设置日期与执行日
4. 确认本次分配对象
5. 提交分配

### 6.2 第一步：选择分配范围

这一块不再先铺满所有人卡片，先让教练决定范围。

建议先提供 4 种模式：

- `整队分配`
- `仅当前未分配`
- `手动点选`
- `整队后剔除`

配套控件：

- 队伍选择
- 项目选择
- 搜索姓名
- 仅看已选
- 清空已选

只有在 `手动点选` 或 `整队后剔除` 模式下，才大面积展示运动员卡片。

### 6.3 运动员卡片改法

当前运动员卡片信息太平均，没有明显主次。

建议卡片只保留：

- 姓名
- 队伍
- 当前是否已有后续计划
- 当前是否已被选中

状态表达建议：

- 已选：高亮底色 + 明显边框
- 已有计划：角标提示 `已有安排`
- 未分配：浅绿角标 `未分配`

不要在卡片里放过多解释性小字。

### 6.4 第二步：选择模板

当前是单个下拉框，能用，但不够清楚。

建议改为：

- 搜索型下拉
- 或左侧列表 + 右侧模板摘要

模板选中后只展示：

- 模板名称
- 模板描述
- 模块数量
- 动作数量
- 是否存在“现场控制”项风险

动作明细先折叠，不在这里占大面积。

### 6.5 第三步：设置日期与执行日

这是当前最需要做语义清理的区域。

建议先加“分配类型”快捷切换：

- `单日计划`
- `日期段计划`
- `每周重复计划`

对应规则：

- 单日计划：只显示一个日期，不显示周几 chips
- 日期段计划：显示开始/结束日期，可选是否每天执行
- 每周重复计划：显示开始/结束日期 + 周几 chips

这样能避免“同一天 + 周一到周日全选”的反直觉表达。

### 6.6 第四步：确认对象

当前右侧预览里的“将分配给以下队员”其实应该提前变成确认步骤。

建议在提交前单独显示：

- 本次对象总人数
- 覆盖哪些队伍
- 已有安排人数
- 未分配人数
- 被手动剔除人数

这一步要更像“确认单”，而不是模板详情页。

## 7. 右侧 sticky 摘要面板方案

右侧面板不再承担“完整模板预览”角色，而改成“摘要 + 风险提示 + 主按钮”。

### 7.1 右侧固定显示内容

- 当前步骤标题
- 已选人数
- 所选模板
- 日期范围
- 执行日摘要
- 风险提示
- 主按钮：`确认分配计划`

### 7.2 动作明细改法

模板动作明细建议改成：

- 默认折叠
- 用户手动点开“查看模板内容”
- 最多显示前 4 到 6 个动作摘要
- 完整详情进入二级抽屉或弹层

这样右侧区域不会持续挤压主流程。

### 7.3 风险提示优先级

右侧风险提示建议按如下优先级展示：

1. 缺少测试结果，训练时需现场控制
2. 选中对象里已有后续计划
3. 当前分配范围为空
4. 时间配置不完整

要让教练先看到会影响训练现场的风险，而不是先看到动作列表。

## 8. 视觉风格建议

### 8.1 整体风格

建议保持当前管理端风格基础，不重做品牌方向。  
重点是“减重、提层级、少留白浪费、少重复盒子”。

### 8.2 视觉规则

- 页面主标题保留强对比
- 摘要卡变矮、信息更集中
- 卡片圆角统一
- 一级按钮只保留一个强调色
- 危险按钮只在取消分配区出现
- 次要说明文字统一收敛为一种灰度

### 8.3 间距与密度

推荐调整：

- 首屏顶部摘要区高度下降约 30% 到 40%
- 运动员卡片高度下降约 15% 到 20%
- 右侧面板宽度固定，减少内容波动
- 空态区高度压到一屏以内

### 8.4 颜色语义

- 已选：绿色系
- 未分配：浅绿色或青绿色标签
- 已有安排：蓝灰标签
- 即将开始：蓝色
- 进行中：绿色
- 危险取消：红色
- 仅说明性提示：中性灰

不要把“已选”和“危险确认”都做成高饱和红色。

## 9. 前端结构改造建议

推荐把当前大页面拆成更明确的组件。

### 9.1 建议组件拆分

- `AssignmentPageHeader`
- `AssignmentViewSwitch`
- `AssignmentOverviewSummary`
- `AssignmentGroupList`
- `AssignmentGroupDetailPanel`
- `AssignmentBuilderScopePanel`
- `AssignmentBuilderTemplatePanel`
- `AssignmentBuilderSchedulePanel`
- `AssignmentBuilderConfirmPanel`
- `AssignmentBuilderSidebar`

### 9.2 建议状态拆分

当前最先应拆开的状态：

- `overviewTargetDate`
- `builderStartDate`
- `builderEndDate`
- `overviewFilters`
- `builderFilters`

不要继续复用同一个 `form.start_date` 同时控制概览和新建表单。

### 9.3 推荐前端实现顺序

1. 先拆状态
2. 再做“现有分配 / 新建分配”视图切换
3. 再压缩现有分配空态与重复列表
4. 再重排新建分配步骤
5. 最后收口样式和 sticky 摘要

## 10. 可选的少量后端补充

当前后端接口整体已经够支撑第一轮 UI 改版。  
如果要进一步提升页面表达，可以补充少量只读字段。

### 10.1 `/api/assignments/overview` 建议补充

建议新增可选只读字段：

- `target_date`
- `next_occurrence_date`
- `team_names`
- `athlete_count_by_team`

用途：

- 更明确地展示“当前在看哪一天”
- 在计划组卡片中直接显示“下次执行”
- 不展开名单也能看出这组主要覆盖哪些队

### 10.2 `/api/assignments/preview` 建议补充

建议新增可选只读字段：

- `athlete_count`
- `manual_control_athlete_count`
- `manual_control_item_count`
- `schedule_mode_hint`

用途：

- 右侧摘要面板更容易直接展示风险和数量
- 前端不用自己扫描所有 rows 才能做基础摘要

### 10.3 不建议做的后端改动

- 不建议改数据库结构
- 不建议改分配写库规则
- 不建议在本轮引入复杂冲突检测
- 不建议新增大量批量编辑接口

## 11. 分阶段实施方案

### Phase A：只做前端结构收口

目标：

- 把“现有分配”和“新建分配”拆开
- 拆开概览和新建表单状态
- 压缩空态和重复列表

特点：

- 不依赖后端改动
- 风险最低
- 最快出效果

### Phase B：做新建分配主流程重排

目标：

- 引入范围选择模式
- 调整步骤顺序
- 右侧预览改成摘要

特点：

- 主要是前端改造
- 对教练使用体验提升最大

### Phase C：补少量只读字段

目标：

- 让现有分配卡片和右侧摘要更完整

特点：

- 后端只加返回字段
- 不改业务写入逻辑

### Phase D：样式与细节收口

目标：

- 搜索、批量选择、筛选条、折叠展开、按钮层级收口

特点：

- 以体验打磨为主

## 12. 验收标准

### 12.1 教练侧人工验收

至少验证下面场景：

1. 能在首屏明确区分“我是在看现有分配”还是“我在新建分配”
2. 能按队伍快速给整队分配模板
3. 能只给当前未分配队员分配
4. 能看清本次分配的人数、模板、日期和执行日
5. 右侧不会长期被动作明细撑满
6. 取消分配入口明确，但不会误触
7. 1366x768 笔记本下主流程不需要频繁上下大滚动
8. iPad 横屏查看时布局仍然清楚

### 12.2 技术验收

- 前端 `npm run build` 通过
- 页面切换不报错
- 概览筛选与新建表单状态不再互相污染
- 预览生成逻辑与提交逻辑保持一致
- 取消分配仍保留二次确认

---

## 13. 最终建议

如果只做一轮最值钱的改动，建议优先顺序如下：

1. 先拆“现有分配 / 新建分配”双视图
2. 再拆概览状态和新建表单状态
3. 再把新建流程改成“范围 -> 模板 -> 时间 -> 确认”
4. 最后把右侧预览收敛成 sticky 摘要

这四步完成后，页面就会从“一个混合功能页”变成“一个教练可连续操作的分配工具页”。
