# 第一阶段总验收清单

> 文档状态：历史阶段文档  
> 当前用途：保留第一阶段收口时的验收参考，不作为当前服务器阶段的唯一总验收标准。  
> 当前优先阅读：`README.md`、`CURRENT_STATUS.md`、部署文档与现有模块专项文档。

## 1. 文档用途

这份文档只服务一件事：

- 把第一阶段是否可以收口，写成一套**按顺序执行、非程序员也能照做**的验收清单。

它不是泛泛的“测试建议”，而是实际验收手册。  
你可以按本文从上到下执行，并记录每一项是否通过。

本清单重点覆盖第一阶段核心主链：

1. 全新电脑初始化与一键启动
2. `1 台电脑 + 2 台 iPad` 的基本训练场景
3. 本地草稿保存、恢复、继续录课
4. 单组增量补传
5. 整课覆盖同步兜底
6. `manual_retry_required` / 待人工处理闭环
7. 启动后跨日收口 + 显式维护 / 非 GET 训练入口兜底收口
8. 训练状态流
9. 课后补录 / 修改 / 删除后的日志
10. 危险操作前备份与日志
11. 迁移命令、备份命令、启动器摘要

## 2. 验收前准备

### 2.1 设备和角色

请准备下面三台设备：

| 设备 | 用途 |
| --- | --- |
| 电脑（Host） | 启动前后端、进入管理模式、查看训练数据/备份/日志 |
| iPad A | 主训练设备，负责真实录组、断网、恢复、手动同步 |
| iPad B | 第二台平板，用来验证同网络访问、旁路观察同步结果 |

### 2.2 测试数据准备

开始前，先准备好：

1. 至少 1 名运动员
2. 至少 1 份当日可命中的训练计划
3. 该计划至少有 **2 个动作**
4. 每个动作至少 **2 组**

这样才能完整覆盖：

- `not_started`
- `in_progress`
- `completed`
- `partial_complete`
- 同步、补录、删除、重算

### 2.3 网络准备

1. 电脑、iPad A、iPad B 先连到同一个 Wi-Fi。
2. iPad 使用 **Safari** 打开系统。
3. 启动后以 `start_system.bat` 窗口里显示的 `Network` 地址为准。

### 2.4 风险控制

在做下面这些验收前，建议先做一次手动备份：

- 跨日收口人工构造
- 训练记录删除
- 备份恢复演练

建议命令：

```powershell
cd backend
.\.venv\Scripts\python.exe scripts\create_backup.py phase1_acceptance
```

### 2.5 验收时建议同时打开的页面

为了减少来回找页面，建议电脑端先打开这些入口：

1. `管理模式 -> 总览`
2. `管理模式 -> 训练数据`
3. `管理模式 -> 日志`
4. `管理模式 -> 备份恢复`

### 2.6 验收记录建议

每做完一项，至少记录：

- 测试编号
- 是否通过
- 失败现象
- 是否阻塞进入第二阶段

## 3. 最低通过标准

只要下面任意一条失败，就**不能进入第二阶段**：

1. 新电脑不能按步骤初始化 / 启动
2. 启动器失败时没有清晰摘要
3. 训练现场断网后不能继续录入
4. 本地草稿丢失或恢复后重复建课
5. 增量补传和整课兜底都不能把本地草稿补回后端
6. `manual_retry_required` 对教练 / 管理端不可见
7. 启动后跨日收口失效
8. 显式维护 / 非 GET 训练入口兜底收口失效
9. `not_started / in_progress / completed / absent / partial_complete` 任一状态流不符合预期
10. 课后修改 / 补录 / 删除后不重算状态或无日志
11. 危险操作前没有自动备份或没有日志
12. 迁移命令或备份命令不可用

---

## 4. 建议验收顺序

不要乱跳。按下面顺序做，最容易尽早暴露第一阶段残留问题：

1. `A01` 全新电脑首次初始化启动
2. `A02` 一键启动 + 1 台电脑 + 2 台 iPad 基本连通
3. `A03` 正常开课，验证 `not_started -> in_progress -> completed`
4. `A04` 本地草稿保存、恢复、继续录课
5. `A05` 单组增量补传
6. `A06` 增量失败后整课覆盖同步兜底
7. `A07` 自动重试失败后进入 `manual_retry_required`
8. `A08` 启动后跨日收口
9. `A09` 显式维护 / 非 GET 入口兜底收口
10. `A10` 手动结束未录完课程，验证 `partial_complete`
11. `A11` 课后补录 / 修改 / 删除 + 日志
12. `A12` 危险操作前备份与日志
13. `A13` 迁移命令、备份命令、健康检查
14. `A14` 启动器摘要与异常提示可读性

## 5. 人工验收清单

---

### A01｜全新电脑首次初始化启动

**测试目标**

- 验证非程序员在一台新电脑上，能按启动器完成首次初始化。
- 验证缺环境或出错时，启动器会给出可复制摘要，而不是黑盒报错。

**前置条件**

- 最好在一台未跑过本项目的新电脑或临时测试电脑上执行。
- 仓库代码已拉到本机。

**测试步骤**

1. 双击 `scripts/init_system.bat`。
2. 观察输出顺序是否是：
   - 检查环境
   - 准备后端环境
   - 准备前端依赖
   - 检查数据库和迁移
3. 初始化完成后，双击 `scripts/start_system.bat`。
4. 如果中途失败，直接打开：
   - `logs/startup/last-launcher-summary.txt`
   - `logs/startup/last-launcher-detail.log`
5. 看摘要里是否能读懂：
   - 失败步骤
   - 错误类型
   - 最可能原因
   - 建议修复

**预期结果**

- 成功时，初始化和启动流程顺序清晰。
- 失败时，摘要文件可直接发给 AI 或维护者，不需要发整屏终端。

**不通过时意味着什么**

- 启动器对非程序员仍然不可用。
- 第一阶段的“可启动、可定位问题”目标未达成。

---

### A02｜一键启动链路 + 1 台电脑 + 2 台 iPad 基本场景

**测试目标**

- 验证正常启动后，电脑、iPad A、iPad B 都能进入系统。
- 验证网页上显示的推荐地址和 `start_system.bat` 的 `Network` 地址一致可用。

**前置条件**

- 电脑已经能跑通 `scripts/start_system.bat`。

**测试步骤**

1. 在电脑上双击 `scripts/start_system.bat`。
2. 记下 BAT 窗口里的 `Network` 地址。
3. 在 iPad A 的 Safari 输入这个地址。
4. 在 iPad B 的 Safari 输入同一个地址。
5. 三台设备分别确认：
   - 电脑能看到训练模式或管理模式
   - iPad A 能打开训练模式
   - iPad B 也能打开同一系统
6. 在电脑点击右上角切到 `管理模式`，确认左侧能看到：
   - `总览`
   - `训练数据`
   - `备份恢复`
   - `日志`

**预期结果**

- 三台设备都能打开系统。
- 地址以 BAT 窗口显示的 `Network` 为准。
- iPad 不需要截图或额外配置就能访问。

**不通过时意味着什么**

- 第一阶段当时定义的本地训练验收链路还不稳定。
- 训练现场部署仍存在入口不一致问题。

---

### A03｜正常开课，验证 `not_started -> in_progress -> completed`

**测试目标**

- 验证训练主链路和三种最基础状态：
  - `not_started`
  - `in_progress`
  - `completed`

**前置条件**

- 某名运动员当天有可用计划。
- 该计划至少 2 个动作、每个动作至少 2 组。

**测试步骤**

1. 在 iPad A 打开训练模式。
2. 选择训练日期、队伍、运动员。
3. 在左侧计划卡片里点击一份计划进入训练课。
4. **不要先录组**，直接点浏览器返回上一页，回到训练模式列表。
5. 观察该计划卡片：
   - 仍应显示为红色 / `未开始`
6. 再次进入这堂训练课。
7. 录入第一组，点击 `确认提交当前组`。
8. 观察状态：
   - 训练页状态应变为 `进行中`
   - 或回到训练模式列表后，计划卡片变为黄色 / `进行中`
9. 继续录完所有动作和所有组。
10. 观察：
    - 每个动作录完会自动回动作列表
    - 全部录完后整堂课变为 `已完成`
11. 在电脑端进入 `管理模式 -> 训练数据`，选择该运动员，展开最新课次卡片。

**预期结果**

- 打开计划但不录组时，仍是 `not_started`
- 第一组成功后才进入 `in_progress`
- 全部录满后进入 `completed`
- 电脑端能看到完整组数和已完成状态

**不通过时意味着什么**

- 训练状态流仍不清晰
- “打开计划不算开始”“第一组成功才正式开始”这条主规则没守住

---

### A04｜本地草稿保存、恢复提示、继续录课

**测试目标**

- 验证训练中断后能从本地草稿恢复。
- 验证恢复后回到动作列表，不创建重复 `session`。

**前置条件**

- iPad A 已经开始一堂训练课，并至少录了 1 组。

**测试步骤**

1. 在 iPad A 录完 1 组或 2 组后，直接关闭当前 Safari 标签页。
2. 重新打开 Safari，再次进入训练模式。
3. 观察是否弹出恢复提示：
   - 标题应类似“检测到本机有未完成训练草稿”
4. 先点击 `继续录课`。
5. 确认恢复后回到原训练课的**动作列表**，而不是随机页面。
6. 再继续录入 1 组，确认可以正常接着录。
7. 重新做一次，第二次在恢复弹层点击 `放弃草稿`。
8. 确认这次草稿被删除，不再反复弹恢复提示。

**预期结果**

- 恢复提示可见
- `继续录课` 后回到原训练课动作列表
- 不会创建重复 `session`
- `放弃草稿` 后，本地草稿消失

**不通过时意味着什么**

- 本地草稿恢复链路不稳定
- 队员重开页面后仍然可能迷路或重复建课

---

### A05｜单组增量补传

**测试目标**

- 验证联网状态下，每提交 1 组，后端很快收到新增记录。

**前置条件**

- 电脑端服务正常
- iPad A、电脑都已打开系统

**测试步骤**

1. 在 iPad A 打开一堂新的训练课。
2. 只录入 1 组，点击 `确认提交当前组`。
3. 不要再录第二组，等待 5～10 秒。
4. 电脑端进入 `管理模式 -> 训练数据`。
5. 选择同一运动员，展开最新训练课卡片。
6. 查看刚刚那个动作的组表。

**预期结果**

- 电脑端能看到新提交的那 1 组
- 组数、重量、次数、RIR 与 iPad A 一致
- 训练端状态灯保持 `正常`

**不通过时意味着什么**

- 单组增量补传链路不稳定
- 后续断网/重试时更容易放大问题

---

### A06｜增量补传失败后，整课覆盖同步兜底

**测试目标**

- 验证增量补传失败时，本地草稿仍是主数据源。
- 验证点击 `同步个人训练数据` 后，整课内容能覆盖回后端。

**前置条件**

- iPad A 正在一堂未完成训练课里。

**测试步骤**

1. 在 iPad A 先正常录入 1 组。
2. 回到电脑，关闭**后端那个 CMD 窗口**，让接口不可用。
3. 回到 iPad A，再继续录入至少 2 组。
4. 观察训练端：
   - 不应被打断
   - 状态灯应变成 `有未同步数据`
5. 电脑重新双击 `scripts/start_system.bat`，把服务拉起来。
6. 回到 iPad A，点击页面上的 `同步个人训练数据`。
7. 电脑端进入 `管理模式 -> 训练数据`，检查该训练课。

**预期结果**

- 断服期间 iPad A 仍能继续录
- 点击 `同步个人训练数据` 后，后端结果与本地草稿一致
- 没有丢组、乱序或少一组

**不通过时意味着什么**

- 第一阶段的整课兜底没有收口完成
- 本地优先原则没有真正落地

---

### A07｜自动重试失败后进入 `manual_retry_required`，并可人工重试

**测试目标**

- 验证自动重试不会无限重试
- 验证失败足够久后会进入 `manual_retry_required`
- 验证训练端和教练端都能手动重试

**前置条件**

- iPad A 已经在一堂训练课里
- 电脑当前服务正常

**测试步骤**

1. 在 iPad A 先录入 1 组，确保当前训练课已经正式开始。
2. 回到电脑，关闭后端 CMD 窗口。
3. 在 iPad A 再录入 1 组。
4. 保持页面停留在训练课里，等待 **20～30 秒**。
5. 观察训练课页面：
   - 顶部应出现类似“自动重试已暂停，需教练手动补传。”
   - 同步状态应不再回到 `正常`
6. 电脑重新启动服务。
7. 在 iPad A 点击 `同步个人训练数据`，确认能恢复。
8. 再做一轮：如果 iPad A 不方便操作，改到电脑端 `管理模式 -> 总览` 或 `管理模式 -> 训练数据`，在 `同步异常待处理` 卡片上点击 `手动重试`。

**预期结果**

- 自动重试失败后，状态进入 `manual_retry_required`
- 训练端不崩，当前浏览器本机草稿仍可恢复
- iPad A 和电脑端都能看到人工处理入口
- 重试成功后，待处理状态消失

**不通过时意味着什么**

- 同步异常闭环还没收住
- 教练收平板时无法知道哪台设备/哪堂课有问题

---

### A08｜启动后跨日收口

**测试目标**

- 验证后端启动时会先自动纠正昨天及更早未收口训练课
- 验证“不进入训练流程，直接先看别的页面或查库”也能看到正确状态

**前置条件**

- 已经有一堂训练课，且你知道它的 `session_id`
- 如果不知道 `session_id`，先在电脑 PowerShell 执行：

```powershell
cd backend
@'
import sqlite3
conn = sqlite3.connect("training.db")
rows = conn.execute("""
SELECT ts.id, ts.session_date, ts.status, COUNT(sr.id) AS recorded_sets
FROM training_sessions ts
LEFT JOIN training_session_items tsi ON tsi.session_id = ts.id
LEFT JOIN set_records sr ON sr.session_item_id = tsi.id
GROUP BY ts.id
ORDER BY ts.id DESC
LIMIT 10
""").fetchall()
for row in rows:
    print(row)
'@ | .\.venv\Scripts\python.exe -
```

**测试步骤**

1. 先做一份手动备份：

```powershell
cd backend
.\.venv\Scripts\python.exe scripts\create_backup.py before_startup_due_close
```

2. 关闭后端服务。
3. 把一堂已有训练课改造成“昨天未收口、且 0 组记录”的状态。把下面命令里的 `SESSION_ID = 你的ID` 改掉再执行：

```powershell
cd backend
@'
import sqlite3
SESSION_ID = 1
conn = sqlite3.connect("training.db")
cur = conn.cursor()
cur.execute("DELETE FROM set_records WHERE session_item_id IN (SELECT id FROM training_session_items WHERE session_id = ?)", (SESSION_ID,))
cur.execute("UPDATE training_session_items SET status = 'pending' WHERE session_id = ?", (SESSION_ID,))
cur.execute("UPDATE training_sessions SET session_date = date('now','-1 day'), status = 'in_progress', started_at = NULL, completed_at = NULL WHERE id = ?", (SESSION_ID,))
conn.commit()
print('mutated session', SESSION_ID)
'@ | .\.venv\Scripts\python.exe -
```

4. 重新启动系统。
5. **不要先进入训练模式。**
6. 直接在电脑 PowerShell 查这堂课：

```powershell
cd backend
@'
import sqlite3
SESSION_ID = 1
conn = sqlite3.connect("training.db")
print(conn.execute("SELECT id, session_date, status, completed_at FROM training_sessions WHERE id = ?", (SESSION_ID,)).fetchone())
'@ | .\.venv\Scripts\python.exe -
```

7. 也可以顺手去 `管理模式 -> 训练数据` 看这堂课的状态。

**预期结果**

- 不需要先打开训练流程
- 后端启动后，这堂课已经被改成 `absent`

**不通过时意味着什么**

- “启动成功即收口成功”这条第一阶段收尾规则没有真正落地
- 报表、日志、管理页仍可能先看到错误口径

### A09｜显式维护与非 GET 入口兜底收口仍有效

**测试目标**

- 验证就算启动后没有触发到，也能通过显式维护接口或非 GET 训练入口兜底收口
- 验证训练端 / 监控端 GET 只读页面不会因为展示数据而隐式写库

**前置条件**

- 系统已经处于运行中
- 已经有一堂**昨天**的、非终态、且至少有 1 组记录的训练课

**测试步骤**

1. 先找一堂已有训练课 `session_id`。
2. 把它改成昨天未收口，但保留至少 1 组记录。把命令里的 `SESSION_ID` 改掉：

```powershell
cd backend
@'
import sqlite3
SESSION_ID = 1
conn = sqlite3.connect("training.db")
cur = conn.cursor()
cur.execute("UPDATE training_sessions SET session_date = date('now','-1 day'), status = 'in_progress', completed_at = NULL WHERE id = ?", (SESSION_ID,))
conn.commit()
print('mutated session', SESSION_ID)
'@ | .\.venv\Scripts\python.exe -
```

3. **不要重启后端。**
4. 先只刷新训练模式首页或监控端页面，确认只读页面不会触发状态变化。
5. 登录后选择该队员对应计划并进入计划详情，触发非 GET 开课入口；或由管理员调用 `POST /api/system/maintenance/close-due-sessions` 显式维护接口。
6. 回到电脑 PowerShell，再查一次这堂课状态：

```powershell
cd backend
@'
import sqlite3
SESSION_ID = 1
```

### A10｜手动结束未录完课程，验证 `partial_complete`

**测试目标**

- 验证“结束计划”不会无脑写成已完成
- 验证已做部分内容但未录满时，状态应为 `partial_complete`

**前置条件**

- iPad A 正在一堂新训练课里
- 这堂课至少还有 1 个动作或 1 组没录

**测试步骤**

1. 在 iPad A 进入训练课。
2. 只录一部分，例如：
   - 第一个动作录满
   - 第二个动作一组都不录
3. 点击右侧按钮 `结束计划`。
4. 在弹窗里确认结束。
5. 观察页面提示。
6. 再去电脑端 `管理模式 -> 训练数据` 查看这堂课。

**预期结果**

- 页面提示应类似“本堂训练已结束，状态记为未完全完成”
- 电脑端对应训练课状态为 `partial_complete`

**不通过时意味着什么**

- 第一阶段状态流仍然混乱
- 结束计划逻辑仍可能误导为 `completed`

### A11｜课后补录 / 修改 / 删除训练组后的日志

**测试目标**

- 验证管理端课后修正闭环：
  - 修改一组
  - 补录一组
  - 删除一组
  - 状态重算
  - 课后修改日志

**前置条件**

- 电脑端进入 `管理模式 -> 训练数据`
- 已有一堂可以展开的训练课

**测试步骤**

1. 在 `训练数据` 页展开一堂训练课。
2. 找到一条组记录，点击 `修改`。
3. 修改重量或次数或 `RIR`，点击 `保存修正`。
4. 观察顶部提示是否为“课后修正已保存，训练课状态已重算。”
5. 找到一个还没满组数的动作，点击 `补录一组`。
6. 填写数据后点击 `确认补录`。
7. 观察顶部提示是否为“漏录组已补录，训练课状态已重算。”
8. 再找一条组记录，点击 `删除`，并确认删除。
9. 观察训练课底部是否出现 `课后修改日志`。
10. 进入 `管理模式 -> 日志`：
    - 按对象类型筛 `训练数据`
    - 按动作类型筛 `课后修改`

**预期结果**

- 修改、补录、删除都能成功
- 每次操作后训练课状态会自动重算
- 训练课卡片里能看到 `课后修改日志`
- 日志页能查到对应的课后修改记录

**不通过时意味着什么**

- 教练课后修正链路还不可靠
- 数据可追溯性不达标

### A12｜危险操作前备份与日志

**测试目标**

- 验证危险操作前会自动备份
- 验证危险操作会写入日志

**测试步骤**

## 6. 轻量技术自检

这部分是**辅助**，不是替代人工验收。

### 6.1 一键自检脚本

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File scripts\phase1_acceptance_check.ps1
```

### 6.2 这个脚本解决什么

它会用最小代价帮你先挡掉几类明显问题：

1. 启动器初始化 smoke
2. 文本编码检查
3. 后端 `compileall`
4. 迁移 `ensure`
5. 后端 Phase 1 smoke
6. 前端 `npm run build`

### 6.3 不能只靠脚本通过的项目

以下内容仍然必须人工看现象：

1. iPad 断网后继续录组
2. 恢复提示弹层与 `继续录课 / 放弃草稿`
3. 训练端状态灯
4. `manual_retry_required` 在训练端和教练端的可见性
5. 两台 iPad 同网访问
6. 课后补录 / 删除时的实际交互

## 7. 哪些问题必须先修，哪些可带入第二阶段

### 7.1 必须先修

下面这些一旦失败，必须先修，不能带入第二阶段：

1. 数据丢失
2. 草稿恢复失败或重复建课
3. 断网 / 断服后无法继续录课
4. `manual_retry_required` 不可见或无法人工重试
5. 启动后跨日收口失效
6. 显式维护 / 非 GET 训练入口兜底收口失效
7. 状态流错乱
8. 危险操作无备份或无日志
9. 迁移 / 备份命令不可用
10. 启动器摘要不可读

### 7.2 可带入第二阶段

下面这些可以记录后带入第二阶段，不阻断第一阶段收口：

1. 页面排版还不够紧凑
2. 图表样式不够漂亮
3. 日志筛选还不够高级
4. 同步冲突界面不够细
5. 备份恢复页还不够高级
6. 权限颗粒度还不够细
