# 第一阶段总验收｜人工清单、自测脚本与放行标准

## 1. 这一步只解决什么

本文件只收口第一阶段“总验收材料”，目标是：

1. 把第一阶段是否通过写成明确标准，而不是靠感觉判断。
2. 给出 **1 台电脑 + 2 台 iPad** 的人工验收路径。
3. 给出一套最小技术自测入口，覆盖第一阶段必须守住的核心链路。
4. 明确哪些不过关就 **不能进入第二阶段**。

本步不展开：

- 第二阶段功能设计
- 复杂自动化 E2E 平台
- 新功能扩展

---

## 2. 验收拓扑

### 2.1 设备角色

- **电脑（Host）**
  - 负责启动前后端
  - 负责教练端 / 管理端回看
  - 负责查看日志、备份、恢复、同步异常
- **iPad A（主训练设备）**
  - 负责队员现场录入
  - 负责验证本地草稿、断网继续录、手动同步
- **iPad B（辅助设备）**
  - 负责验证同网络访问
  - 负责验证另一台设备/教练端能否看到同步后的结果

### 2.2 基础前置条件

总验收前，至少准备：

1. 一名有当日可用计划的运动员。
2. 一份包含 **至少 2 个动作、每个动作至少 2 组** 的训练模板。
3. 电脑和 2 台 iPad 在同一局域网内。
4. 电脑端已具备可用的 `start_system.bat` / `init_system.bat` 环境。
5. iPad 使用 Safari 打开训练端。

### 2.3 验收记录建议

每条验收都记录：

- 测试人
- 测试日期
- 使用设备
- 结果：`通过 / 失败 / 阻塞`
- 失败现象
- 是否阻断第二阶段

---

## 3. 阻塞放行标准

只要以下任一阻塞项失败，**第一阶段不得判定通过，也不得进入第二阶段**。

| 阻塞项 | 必须通过的结果 |
| --- | --- |
| G01 | 全新电脑能按启动器完成初始化，失败时能给出可复制摘要 |
| G02 | 正常开课、正常录完整堂课，全流程不丢数据 |
| G03 | iPad 断网或电脑端服务短时断开后，训练端仍可继续录入，本地草稿不丢 |
| G04 | 恢复网络/服务后，能完成自动或人工同步；同步异常对教练/管理员可见 |
| G05 | 本地草稿恢复后，不创建重复 `session`，且能回动作列表继续 |
| G06 | 增量补传失败后，整课覆盖同步兜底可用；明显冲突有提示/日志 |
| G07 | 课后修改、补录、状态重算、修改日志链路可用 |
| G08 | 迁移、备份、恢复基础能力可用 |
| G09 | `scripts/phase1_acceptance_check.ps1` 返回 `0` |

## 4. 人工验收清单（1 台电脑 + 2 台 iPad）

下面每条都包含：目标、前置条件、操作步骤、通过标准、失败定义、是否阻断第二阶段。

### M01｜全新电脑首次初始化启动

**目标**

- 验证非程序员在新电脑上也能按步骤初始化并启动。

**前置条件**

- 电脑未提前手工配置项目依赖，或至少模拟为“首次运行”场景。

**步骤**

1. 在电脑上执行 `scripts/init_system.bat`。
2. 观察是否按“检查环境 -> 安装小依赖/初始化 -> 数据库迁移检查”顺序输出。
3. 再执行 `scripts/start_system.bat`。
4. 记录前端访问地址、后端是否启动、失败时是否生成摘要。

**通过标准**

- 初始化流程顺序清晰。
- 缺环境时有明确的失败步骤、错误类型、最可能原因、建议修复。
- 成功时前后端可访问。
- 失败时存在可复制摘要和详细日志路径。

**失败定义**

- 只出现黑盒报错，没有摘要。
- 用户无法判断卡在哪一步。
- 启动器反复执行后结果不一致或把环境弄坏。

**是否阻断第二阶段**

- 是。

### M02｜正常开课并完整录完一堂课

**目标**

- 验证训练主链路：模板 -> 计划分配 -> 当天训练执行 -> 组记录 -> 完课。

**前置条件**

- 电脑端服务正常。
- iPad A 打开训练端。
- 至少一名运动员有当日有效计划。

**步骤**

1. iPad A 进入训练模式，选择运动员和当日计划。
2. 只打开计划，不录组，确认此时不算正式开始。
3. 进入第一个动作，提交第一组。
4. 继续录完该动作，确认会自动标记完成并回动作列表。
5. 录完整堂课所有动作。
6. 在电脑端训练回看页确认训练课状态。

**通过标准**

- 打开计划不创建正式训练开始状态。
- 第一组提交成功后才进入 `in_progress`。
- 动作录完自动回动作列表。
- 整堂课录完后最终状态为 `completed`。
- 电脑端/教练端能看到完整组记录。

**失败定义**

- 打开计划就提前创建正式训练课。
- 录组过程中跳转混乱，队员迷路。
- 录满后状态不是 `completed`。
- 数据丢失或记录数量不一致。

**是否阻断第二阶段**

- 是。

### M03｜iPad 训练中断网后继续录入，稍后恢复

**目标**

- 验证训练现场断网时仍可继续录入，本地草稿和后续恢复可用。

**前置条件**

- iPad A 已经开始一堂训练课，且已成功录入至少 1 组。

**步骤**

1. 在 iPad A 上关闭 Wi-Fi，或让其脱离当前局域网。
2. 继续录入多个动作/多组数据。
3. 观察训练端是否仍能继续录，不应被同步失败打断。
4. 恢复网络。
5. 等待自动同步，若未恢复则手动点击“同步个人训练数据”。
6. 在电脑端或 iPad B 查看这堂课是否同步回来。

**通过标准**

- 断网期间仍能继续录入。
- 断网期间数据先留在本地草稿。
- 恢复网络后能自动或手动同步成功。
- 恢复后后端记录与本地草稿一致。

**失败定义**

- 断网直接中断训练录入。
- 恢复网络后丢组或顺序错误。
- 队员需要处理复杂同步异常才能继续训练。

**是否阻断第二阶段**

- 是。

### M04｜电脑端服务重启/断开后，iPad 本地继续录

**目标**

- 验证即使 Host 端服务短时不可用，iPad 仍能继续录课。

**前置条件**

- 电脑端已经启动服务。
- iPad A 正在一堂未完成训练课里。

**步骤**

1. 在电脑上关闭后端/前端服务窗口，或模拟服务不可达。
2. iPad A 继续录入至少 2 组。
3. 确认训练端没有因为接口失败而卡死。
4. 电脑重新启动服务。
5. 回到 iPad A，等待自动重试或手动点击“同步个人训练数据”。
6. 在电脑端 / iPad B 查看同步结果。

**通过标准**

- 服务不可用时，iPad A 仍可继续录入。
- 草稿保留完整。
- 服务恢复后，这些记录能补回后端。
- 同步异常期间，教练端能看到“待处理”或恢复后的结果。

**失败定义**

- 服务断开导致训练端无法继续。
- 服务恢复后，本地记录无法追回。
- 教练端完全看不到异常或恢复状态。

**是否阻断第二阶段**

- 是。

### M05｜设备重开后的草稿恢复与继续录课

**目标**

- 验证重开 Safari / 重进页面后，用户不会迷路，也不会生成重复 `session`。

**前置条件**

- iPad A 上有一堂未完成训练课，且至少录过 1 组。

**步骤**

1. 关闭当前 Safari 标签页，或刷新训练端页面。
2. 重新进入训练模式。
3. 观察是否出现恢复提示。
4. 选择“继续录课”。
5. 确认恢复后优先回到动作列表。
6. 继续进入某个动作继续录组。
7. 再做一轮，选择“放弃草稿”。

**通过标准**

- 会出现恢复提示。
- 选择继续后，回到原训练课动作列表。
- 不会新建重复 `session`。
- 选择放弃后，本地草稿被删除。

**失败定义**

- 重进后没有恢复提示。
- 继续后直接迷路到错误页面。
- 恢复过程产生重复训练课记录。

**是否阻断第二阶段**

- 是。

### M06｜同步状态灯、异常待处理与人工重试

**目标**

- 验证同步状态对教练可见，但不要求队员理解。

**前置条件**

- 已制造一堂 `pending / manual_retry_required` 的训练课。

**步骤**

1. 在 iPad A 上制造未同步数据。
2. 观察训练端状态灯是否从“正常”切到“有未同步数据”。
3. 在电脑端训练回看页或管理端查看是否出现“同步异常，待处理”。
4. 分别在 iPad A 与电脑端执行一次手动重试。

**通过标准**

- 训练端灯可见，状态与实际同步情况一致。
- 教练端 / 管理端能看到待处理异常。
- 手动重试成功后，异常状态消失或转 resolved。

**失败定义**

- 队员端和教练端状态不一致。
- 发生异常但教练/管理员完全不可见。
- 没有人工重试入口。

**是否阻断第二阶段**

- 是。

### M07｜整课覆盖同步兜底与冲突提示

**目标**

- 验证增量补传不稳时，整课同步能以本地草稿为准兜底。

**前置条件**

- 已有一堂存在本地草稿且后端未完全同步的训练课。

**步骤**

1. 在 iPad A 上制造多次增量失败，或直接手动触发整课同步。
2. 确认后端最终结果与本地草稿一致。
3. 在同步前，若可能，在电脑端对同一堂课制造后端变化。
4. 再次执行整课同步。
5. 在教练端 / 日志页确认冲突提示或冲突日志。

**通过标准**

- 整课同步后，后端结果以本地草稿为准。
- 明显冲突会被记录，并对教练/管理员可见。
- 训练端不因冲突而被打断。

**失败定义**

- 整课兜底无法恢复本地草稿结果。
- 冲突发生但没有日志或提示。

**是否阻断第二阶段**

- 是。

### M08｜课后修改、补录与状态重算

**目标**

- 验证课后修正不破坏一致性，并且可追溯。

**前置条件**

- 已有一堂已完成或未完全完成的训练课。

**步骤**

1. 在电脑端课后修改一组重量 / 次数 / RIR。
2. 再补录一组遗漏的组记录。
3. 观察训练课状态是否自动重算。
4. 到日志页查看对应修改日志。

**通过标准**

- 修改后数据立即反映在训练课里。
- 补录后状态按最新结果重算。
- 日志能看到时间、操作人、对象、关键改动。

**失败定义**

- 修改后状态不重算。
- 补录后日志缺失。
- 修改造成训练课数据损坏。

**是否阻断第二阶段**

- 是。

### M09｜备份与恢复基础能力

**目标**

- 验证第一阶段的备份、恢复、恢复提示和恢复日志链路。

**前置条件**

- 系统中已有至少一个备份点。

**步骤**

1. 在电脑端查看备份列表。
2. 选一个明确时间点的备份。
3. 执行一次恢复前确认，核对影响范围和确认词。
4. 执行整库恢复或某类数据恢复。
5. 观察恢复完成提示中的时间点。
6. 到日志页查看恢复日志。

**通过标准**

- 能看到备份列表。
- 恢复前有二次确认、影响范围、确认词。
- 恢复后明确显示恢复到了哪个时间点。
- 恢复操作被写入日志。

**失败定义**

- 恢复入口不可用。
- 恢复后不知道恢复到了哪一刻。
- 没有恢复日志。

**是否阻断第二阶段**

- 是。

## 5. 技术自测清单

## 5.1 一键自测入口

第一阶段最小自动自测入口：

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File scripts\phase1_acceptance_check.ps1
```

### 5.2 自测脚本覆盖项

`scripts/phase1_acceptance_check.ps1` 当前固定覆盖：

| 编号 | 检查项 | 覆盖目标 |
| --- | --- | --- |
| C01 | 启动器初始化 smoke | 启动器、失败摘要、依赖初始化入口 |
| C02 | 文本编码检查 | 避免核心文档/脚本编码回归 |
| C03 | 后端 `compileall` | 后端基础语法与导入稳定性 |
| C04 | 数据库迁移 `ensure` | 正式迁移链路可执行 |
| C05 | 后端 Phase 1 smoke | 增量补传、整课兜底、同步异常人工重试、状态重算、备份/恢复基础能力 |
| C06 | 前端 `npm run build` | 前端类型检查与构建稳定性 |

### 5.3 后端 Phase 1 smoke 细项

`backend/scripts/phase1_backend_smoke_check.py` 在 **临时数据库副本** 上执行，不碰正式训练库，当前覆盖：

1. 打开计划只返回预览，不提前创建正式 `session`
2. 第一组提交成功后创建/确认 `session`，状态进入 `in_progress`
3. 将整堂课动作置为完成后，状态重算为 `completed`
4. 用错误的 `last_server_signature` 执行整课覆盖，验证冲突日志落库
5. 上报 `manual_retry_required` 同步异常并执行人工重试，验证异常闭环
6. 在临时库上创建备份、修改数据、再从备份恢复，验证恢复基础能力

### 5.4 浏览器级技术检查（当前未自动化）

以下项目属于第一阶段必须检查，但当前仓库 **没有引入浏览器 E2E 框架**，因此仍需按技术检查步骤手动执行：

#### T01｜本地草稿保存/恢复

1. iPad A 或桌面浏览器进入训练课并录入至少 1 组。
2. 刷新或重开页面。
3. 确认出现恢复提示。
4. 继续录课后回到动作列表，而不是重复新建 `session`。

#### T02｜同步状态灯

1. 断网或停服务制造未同步状态。
2. 确认训练端状态灯切换为“有未同步数据”。
3. 恢复后确认状态回到“正常”。

#### T03｜增量补传

1. iPad A 正常联网录入 1 组。
2. 在电脑端训练回看页确认很快能看到新增组。
3. 记录数量与顺序必须一致。

#### T04｜整课覆盖同步兜底

1. 制造多次增量失败。
2. 手动触发“同步个人训练数据”。
3. 确认后端最终结果与本地草稿一致。

这些浏览器级检查若失败，同样按阻塞项处理，不能因为“脚本已经通过”而放行。

## 6. 通过 / 失败判定

### 6.1 第一阶段通过条件

第一阶段只有在以下条件 **全部满足** 时，才可判定通过：

1. `M01 ~ M09` 中所有阻塞项全部通过。
2. `scripts/phase1_acceptance_check.ps1` 返回 `0`。
3. 没有出现训练现场数据丢失。
4. 没有出现同步失败直接打断训练。
5. 没有出现草稿恢复导致重复 `session`。
6. 没有出现恢复/备份/迁移基础能力不可用。

### 6.2 明确失败定义

以下任一出现，即视为 **第一阶段失败**：

- 训练端断网后无法继续录入
- 训练端本地草稿丢失
- 恢复草稿后创建重复 `session`
- 同步异常对教练/管理员不可见
- 手动重试入口缺失或无效
- 整课覆盖同步无法把本地草稿补回后端
- 课后修改/补录后状态不重算
- 启动器在新电脑上不能给出可操作错误摘要
- 迁移 / 备份 / 恢复任一基础链路不可用

### 6.3 第二阶段放行条件

只要存在任一阻塞失败项：

> **不得进入第二阶段。**

不能以“功能大体可用”“大部分能跑”替代放行标准。
