# 第一阶段重构补充｜同步状态灯与异常待处理闭环

## 1. 这一步只解决什么

本文件只记录第一阶段中“同步状态可见性与人工兜底”这一小步，目标是：

- 训练端继续保持本地优先录入
- 教练收平板时，一眼能看出哪堂课还有未同步数据
- 教练端 / 管理端能看到“同步异常，待处理”
- 自动重试长期失败后，明确停止并转人工

本步**不**展开：

- 复杂冲突合并界面
- 复杂监控图表
- 更细的多设备冲突工作台

当前同步状态闭环依赖训练端浏览器本地草稿（现阶段为 `localStorage` 优先）；它主要保证同一浏览器 / 同一设备内断网续录和恢复，不等同于跨设备永久可恢复。

## 2. 当前最小状态模型

### 训练端状态灯

训练端页面只保留两态显示：

- `正常`
- `有未同步数据`

队员不需要理解更多状态语义；状态灯主要服务于教练收设备时的快速判断。

### 内部同步状态

本步在本地草稿层继续使用更细的内部状态：

- `synced`
- `pending`
- `manual_retry_required`

其中：

- `pending` 表示仍允许后台自动补传
- `manual_retry_required` 表示自动重试已停，需要人工触发

## 3. 后端最小闭环

新增表：`training_sync_issues`

用途：

- 持久化“同步异常，待处理”
- 保存最近一次可用于人工重试的整课草稿快照
- 作为教练回看页和管理端首页的最小可见数据源

最小字段包括：

- `athlete_id`
- `assignment_id`
- `session_id`
- `session_date`
- `session_key`
- `issue_status`
- `summary`
- `failure_count`
- `last_error`
- `sync_payload`
- `resolved_at`

## 4. 进入人工处理的规则

当前规则收口为：

1. 先按组做增量补传
2. 连续失败达到阈值后，进入整课覆盖兜底
3. 如果整课覆盖也失败，则：
   - 停止继续自动重试
   - 本地草稿进入 `manual_retry_required`
   - 后端写入 / 刷新 `training_sync_issues`

## 5. 人工重试入口

本步要求的最小入口已落到三处：

- 训练端设备本身：训练页“整课补传 / 手动重试同步”
- 教练端：训练回看页“同步异常待处理”面板
- 管理端：首页“同步异常待处理”面板

## 6. 成功后的收口规则

人工重试成功后：

- 后端 `training_sync_issues` 标记为已解决
- 教练端 / 管理端待处理列表消失
- 训练端在重新联网后会尝试对齐已解决的异常状态

## 7. 本步边界

这一步仍然没有解决：

- 更细粒度的冲突审核
- 复杂日志工作台
- 设备身份管理
- 多设备同时编辑的高级冲突裁决

这些内容留到后续步骤继续推进。
