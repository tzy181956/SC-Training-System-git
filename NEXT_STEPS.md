# NEXT_STEPS

## 1. 先完成第一阶段总验收

**目标**  
按 `docs/phase1-final-acceptance.md` 把第一阶段核心主链完整跑一遍，并留下通过/失败记录。

**为什么优先做**  
当前最需要的是确认第一阶段是否真的收口，而不是继续靠感觉往后推进。

**预期产出**  

- 一份逐项验收结果
- 一份阻塞问题清单
- 一次 `scripts/phase1_acceptance_check.ps1` 结果

**验收标准**  

- 阻塞项全部通过后，才允许进入第二阶段
- 失败项能明确归类到“必须先修”或“可延后”

## 2. 继续核对真实数据展示结果

**目标**  
确认真实 Excel 导入后的运动员主档、测试记录、时间文本和力量体重比在 UI 上展示正确。

**为什么优先做**  
当前已经进入真实数据验证阶段，数据正确性比继续加功能更重要。

**预期产出**  

- 一轮人工核对结果
- 一份“展示正确 / 展示仍有偏差”的页面清单

**验收标准**  

- 运动员页能看到真实队员数据
- 测试记录页能看到真实导入结果
- `3000m / 17折*4` 时间文本显示正常
- 卧推 / 卧拉 / 深蹲 / 臀桥的力量体重比计算正确

## 3. 固化真实数据导入流程

**目标**  
把“清理旧业务数据并导入真实 Excel”的过程沉淀成稳定、可重复、可复查的步骤。

**为什么优先做**  
当前真实数据已经成为默认工作模式，跨电脑继续开发时必须能重复导入，而不是依赖一次性手工处理。

**预期产出**  

- 更明确的导入说明
- 导入前检查项
- 导入后校验清单

**验收标准**  

- 新电脑或新数据库上可以重复执行导入
- 执行后结果稳定一致
- 导入失败时能定位到具体字段或表头问题

## 4. 持续收口训练模板与计划分配交互

**目标**  
继续把模板编辑和计划分配收成更适合真实工作流的交互。

**为什么优先做**  
这两块已经具备功能，但仍在持续做“减少重复信息、减少误操作、压缩空白”的优化。

**预期产出**  

- 更稳定的模板编辑体验
- 更紧凑、更清晰的计划分配工作流

**验收标准**  

- 教练端不需要反复在多个区域看同一份信息
- 批量分配、取消分配、预览异常提示逻辑一致
- 模板设置与训练执行行为严格一致

## 5. 建立中文文本长期检查习惯

**目标**  
把中文乱码问题从“事后发现”变成“提交前阻断”。

**为什么优先做**  
这次已经确认仓库里曾经发生过内容级乱码，后续必须防止再次出现。

**预期产出**  

- 每轮改动都运行 `scripts/check_text_encoding.py`
- 协作文档保持 UTF-8 可读
- 不再把终端乱码误判成文件损坏

**验收标准**  

- 新提交的文档与源码不再出现典型 mojibake
- 检查脚本可稳定发现异常文本
- 开发流程中已形成固定校验动作

## 6. 评估是否需要正式迁移机制

**目标**  
决定项目是否继续维持当前的 `init_db + schema_sync` 方式，还是引入正式迁移体系。

**为什么优先做**  
最近模型和业务字段仍在持续变化，如果继续扩展，长期维护成本会越来越高。

**预期产出**  

- 一份明确的数据库演进方案

**验收标准**  

- 能清楚回答“新增字段后，旧库如何升级”
- 跨电脑切换时不会因为库结构不一致导致不可用
