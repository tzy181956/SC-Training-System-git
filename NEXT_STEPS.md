# NEXT_STEPS

## 1. 核对真实数据展示结果

**目标**  
确认 Excel 导入后的运动员主档、测试记录、时间文本、力量体重比在 UI 上展示正确。

**为什么优先做**  
当前已经进入真实数据验证阶段，先确认数据正确比继续加功能更重要。

**预期产出**  
- 一轮人工核对结果
- 明确哪些页面展示正确，哪些页面仍有偏差

**验收标准**  
- 运动员页能看到 16 名女篮青年队队员
- 测试记录页能看到 2026-03-14 测试数据
- `3000m / 17折*4` 显示可读时间文本
- 卧推 / 卧拉 / 深蹲 / 臀桥的力量体重比能按体重自动算出

## 2. 清理真实使用路径中的乱码文本

**目标**  
把当前还残留在页面和文档中的乱码中文逐步替换成正常文案。

**为什么优先做**  
现在已经不是纯 demo 阶段，真实数据进入系统后，乱码会直接影响后续继续开发和实际使用。

**预期产出**  
- 一批核心页面文案修复
- 关键文档转为稳定 UTF-8 内容

**验收标准**  
- README / PROJECT_CONTEXT / DEVELOPMENT_GUIDE 至少不影响阅读
- 运动员、测试、训练、计划分配几个主页面不再出现明显乱码

## 3. 固化真实数据导入流程

**目标**  
把“清理旧业务数据并导入真实 Excel”的过程变成稳定、可复用、可复查的步骤。

**为什么优先做**  
你是单人开发，而且要跨电脑继续工作；导入流程必须可重复，不能只靠一次性临时操作。

**预期产出**  
- 导入说明
- 导入脚本使用方式
- 导入后校验清单

**验收标准**  
- 新电脑或新数据库上可以重复执行导入
- 执行后结果稳定得到同样的 16 名队员和对应测试记录
- 导入失败时能定位到具体队员和列

## 4. 重新检查计划分配与训练模式中的“示例假设”

**目标**  
确认从真实数据切换后，计划分配、训练模式、测试数据之间没有继续依赖旧 demo 假设。

**为什么优先做**  
当前不少业务规则最初是围绕示例数据打通的，真实数据进入后容易暴露隐藏假设。

**预期产出**  
- 一份问题清单
- 若干高优先级修复点

**验收标准**  
- 没有因为缺少旧 demo 测试基准而阻塞分配
- 训练模式不会给出错误默认重量
- 真实队员可以正常进入训练计划和测试链路

## 5. 评估是否需要正式迁移机制

**目标**  
决定项目是否继续维持当前的 `init_db + schema_sync` 方式，还是引入正式迁移。

**为什么优先做**  
最近已经开始扩表；如果后续还要持续改模型，不尽快定策略会增加维护成本。

**预期产出**  
- 明确的数据库演进方案

**验收标准**  
- 能清楚回答“新增字段后，旧库怎么升级”
- 跨电脑切换时不会因为库结构不一致导致不可用
