# 项目背景说明

## 项目名称

体能训练管理平台 V1

## 一句话目标

这是一个面向团队体能训练场景的 Web 平台，重点支持教练制定模板、分配计划、在训练中记录数据，并结合测试结果持续回看训练效果。

## 当前产品定位

当前版本不是通用 SaaS，而是面向真实队伍使用的“记录优先”系统。现阶段更强调：

- 真实业务链路能跑通
- 教练和队员能实际使用
- 训练与测试数据能稳定落库
- 页面与数据结构便于后续继续扩展

## 当前技术栈

- 前端：Vue 3
- 后端：FastAPI
- 数据库：SQLite
- ORM：SQLAlchemy
- 图表：ECharts
- 前后端分离

## 当前运行方式

- 启动入口：`scripts/start_system.bat`
- 初始化入口：`scripts/init_system.bat`
- 前端开发端口：`5173`
- 后端开发端口：`8000`

当前系统采用免登录模式：

- 默认进入训练模式
- 右上角切换到管理模式
- 本地记住上次使用的模式

## 当前核心业务链路

1. 管理运动员主档与队伍信息
2. 记录和导入测试数据
3. 维护动作库
4. 维护训练模板
5. 分配计划到队员
6. 在训练模式中执行计划并记录组数据
7. 回看训练与测试数据

## 核心业务对象

- 运动员 `athletes`
- 项目 `sports`
- 队伍 `teams`
- 动作分类 `exercise_categories`
- 动作 `exercises`
- 训练模板 `training_plan_templates`
- 模板动作 `training_plan_template_items`
- 计划分配 `athlete_plan_assignments`
- 训练 session `training_sessions`
- session 动作项 `training_session_items`
- 组记录 `set_records`
- 测试记录 `test_records`

## 动作库模块现状

动作库已从旧的“平面动作 + 平面标签”模式，升级为“分类树 + 结构化标签”模式。

唯一权威数据源：

- Excel：`exos_action_library_tagged_for_codex.xlsx`
- 主数据表：`动作库_标签版`

落库结构：

- `exercise_categories`
  - 一级分类
  - 二级分类
  - 基础动作
- `exercises`
  - 具体动作
  - 英文原名
  - 分类路径
  - 结构化标签
  - 搜索关键词
  - 原始行数据

网页端动作库当前定位为日常维护后台：

- 分类浏览
- 搜索
- 标签筛选
- 新建 / 编辑 / 删除

Excel 导入能力只保留在后端脚本/API，不再放在网页端主流程里。

## 训练模板与计划分配现状

训练模板当前遵循“记录优先、复杂配置延后”的原则：

- 模板动作卡片按卡片顺序作为训练顺序
- 不再暴露顺序输入
- `rest_seconds` 已从前后端和数据库中彻底移除
- 自动调重默认关闭，且未开启时训练中不会生成建议
- 模板统一保存，不再对单个动作卡片单独保存

计划分配当前特点：

- 上半区概览显示“查看日期当天 + 未来有效计划”
- 概览会区分：
  - `进行中`
  - `即将开始`
- 右侧未分配名单定义为“从当前日期往后没有任何有效计划的人”
- 分配预览已改为：
  - 队员名单
  - 单份模板预览
  - 异常提示区

## 训练模式现状

训练模式当前强调：

- 共享 iPad 使用
- 触摸优先
- 大按钮录入
- 当前组确认提交
- 历史组即时修改
- 自动完成动作与 session

训练模式顶部当前支持：

- 显示同网可访问地址
- 一键复制
- 二维码扫码进入

## 真实数据现状

当前本地库 `backend/training.db` 已进入真实数据验证阶段：

- 运动员已使用真实队伍数据
- 测试记录已从真实 Excel 导入
- 模板与动作库保留
- 示例运动员与旧测试记录已清理

真实导入脚本：

- `backend/scripts/import_real_test_data.py`

它当前负责：

- 清理旧运动员/测试/分配/session/组记录
- 导入女篮青年队真实测试 Excel
- 保留训练模板和动作库

## 当前架构取舍

- 仍未引入正式数据库迁移框架
- 数据库结构演进主要依赖：
  - `backend/scripts/init_db.py`
  - `backend/app/core/schema_sync.py`
- 当前不引入复杂权限系统
- 当前不把 AI 能力做成真实在线依赖，只保留业务预留位和规则逻辑

## 当前最高优先级

1. 确保真实数据链路稳定可重复
2. 保持训练模式和计划分配主流程稳定
3. 清理乱码和协作文档问题，降低跨电脑/跨会话接手成本
4. 在不破坏现有链路的前提下继续细化交互和展示

## 当前开发约束

- 不轻易破坏一键启动链路
- 不把关键业务规则写死在前端
- 面向教练与训练端的页面尽量全中文
- 改动后必须同步：
  - `CHANGELOG.md`
  - 必要的协作文档
- 中文文件检查不要只依赖终端输出，需结合编辑器或 UTF-8 读取验证
