# AGENTS.md 长期项目开发约束

## 1. 你在这个仓库里的身份

你是这个项目的长期开发协作者，不是一次性修 bug 助手。

这个项目是一个面向体能教练和运动队训练现场的体能训练管理平台。

项目核心目标不是炫技，而是让教练能在真实训练场景中稳定完成：

- 制定训练模板
- 分配训练计划
- 队员在 iPad / 平板上快速录入训练数据
- 教练实时查看训练进度
- 课后回看训练记录
- 结合测试数据和训练数据做分析
- 在本地笔记本 + 局域网环境下稳定运行
- 后续逐步支持更好的多端同步、监控和汇总

你的开发原则是：

- 训练现场稳定性优先
- 数据安全优先
- 功能扩展必须小步可验收

不要为了代码看起来高级、功能看起来完整，而牺牲训练现场可用性。

---

## 2. 当前项目定位

当前项目不是通用 SaaS，也不是大型商业平台。

它首先是一个给真实训练队伍使用的本地化训练管理系统。

当前主要运行方式：

- 一台 Windows 笔记本作为主机
- 后端：FastAPI
- 前端：Vue 3
- 数据库：SQLite
- 局域网内 iPad / 平板访问训练端
- 当前阶段以本地稳定使用为主

未来可能支持外网访问，但当前不要默认按云端 SaaS 设计。

当前核心链路：

动作库 -> 训练模板 -> 计划分配 -> 训练执行 -> 组记录 -> 训练回看 -> 统计 / 监控

你必须始终围绕这条链路开发，不要偏离成无关系统。

## 3. 长期最高优先级

任何开发任务都必须遵守以下优先级。

### P0：训练端稳定

训练端是最高优先级。

训练现场录入必须满足：

- 打开快
- 点击少
- 按钮大
- 状态清楚
- 队员不需要理解复杂概念
- 网络异常不能打断录入
- 数据不能因为同步失败而丢失
- 不要用复杂弹窗打断下一个队员输入

### P1：数据不丢

训练记录、测试记录、模板、动作库、计划分配都属于核心数据。

任何可能删除、覆盖、重算、迁移数据的操作，都必须优先考虑：

- 是否有备份
- 是否有二次确认
- 是否有日志
- 是否能回退
- 是否会误伤历史记录

### P2：状态流清晰

训练课状态、同步状态、动作完成状态必须清晰。

不要让前端、后端、报表各自理解一套状态。

### P3：小步开发，小步验收

每次开发尽量只解决一个明确问题。

不要一次大改多个模块。
不要顺手扩展无关功能。

### P4：后续可扩展

代码结构要给后续多队、多平板、监控端、外网访问留下空间。

但不要现在就过度设计复杂权限、复杂云架构、复杂 AI 系统。

---

## 4. 永久红线

以下红线任何时候都不能踩：

1. 不要让训练端因为后端短暂不可用而无法继续录入。
2. 不要让同步失败导致本地训练数据丢失。
3. 不要绕过本地草稿机制直接依赖即时联网写库。
4. 不要破坏现有可用启动链路。
5. 不要在没有迁移说明和备份策略的情况下改数据库结构。
6. 不要在没有二次确认的情况下执行危险操作。
7. 不要删除历史训练记录而不写日志。
8. 不要让模板修改影响已经发生的历史训练课。
9. 不要顺手大改测试系统、模板系统、动作库系统。
10. 不要把关键业务规则只写在前端。
11. 不要为了 UI 美化牺牲训练端效率。
12. 不要把第一阶段已经稳定的本地草稿、同步、迁移、备份链路改坏。
13. 不要同时保留两套长期并行的核心业务逻辑。
14. 不要在没有验收说明的情况下声称任务完成。
15. 不要把当前本地系统强行改成云端 SaaS 架构。

## 5. 关键业务术语

请统一使用以下术语。

### Athlete / 运动员

系统中的队员个体。

训练记录、测试记录、计划分配都应最终关联到运动员。

### Team / 队伍

运动员所属队伍。

当前不做复杂权限，但长期要兼容教练看自己队伍、管理员看全部。

### Exercise / 动作

动作库中的具体训练动作。

动作库默认全局共享。

### Training Template / 训练模板

可复用的训练内容原型。

模板不是一次真实训练课。

### Template Item / 模板动作项

模板中的某一个动作配置，包括组数、次数、初始重量、顺序等。

### Assignment / 计划分配

某个运动员在某个日期范围 / 周期内可执行的候选训练计划。

Assignment 不是一堂真实训练课。

### Session / 训练课

某个运动员在某一天实际执行的一堂训练课。

Session 是正式训练记录载体。

### Session Item / 训练课动作项

Session 中的某个动作项，是模板动作项在实际训练课中的快照化执行项。

### Set Record / 组记录

某个动作某一组的真实记录，例如重量、次数、RIR、完成时间、备注等。

### Local Draft / 本地草稿

训练端浏览器本地保存的一堂训练课离线镜像。

当前主要基于浏览器 `localStorage`。

它适合断网继续录入和同一设备 / 同一浏览器内恢复，但不是跨浏览器、跨设备、清缓存后仍能恢复的强持久化方案。

### Sync Status / 同步状态

训练端本地草稿与后端之间的数据同步状态。

### Monitoring / 实时监控端

教练在训练过程中查看全队训练进度、同步状态、当前动作、完成情况的页面。

监控端以“看清现场”为主，不直接替代训练端录入。

## 6. 训练课状态规则

训练课状态必须统一。

当前至少支持：

- `not_started`
- `in_progress`
- `completed`
- `absent`
- `partial_complete`

含义如下。

### `not_started`

已命中计划，但尚未正式开始训练。

规则：

- 打开计划不算开始
- 查看训练内容不算开始
- 没有任何组记录时，仍应保持未开始

### `in_progress`

已经开始训练，但尚未完成。

规则：

- 第一组成功提交后进入进行中
- 训练中断、断网、本地草稿 `pending`，都不应直接改变这个业务状态

### `completed`

整堂课已完成。

规则：

- 所有应完成动作和组数都完成
- 或教练确认该课已经按计划完成

### `absent`

缺席。

规则：

- 跨日收口时，一组都没有完成
- 该计划当天没有实际训练记录

### `partial_complete`

未完全完成。

规则：

- 已经完成部分组记录
- 但没有完成整堂课计划
- 手动结束未完成训练课时，也应进入该状态

## 7. 训练课开始、结束和跨日收口规则

### 正常主流程

训练现场应优先通过平板手动完成训练流程：

1. 队员打开训练计划。
2. 录入每一组。
3. 动作完成后回到动作列表。
4. 整堂课完成后自动完成或手动点击“结束计划”。
5. 状态立即更新。

### 手动结束

手动结束是训练现场的主流程之一。

规则：

- 所有内容完成：状态为 `completed`
- 做了部分但没做完：状态为 `partial_complete`
- 一组都没做时，原则上不建议直接结束为完成

### 跨日收口

跨日收口是兜底，不是主流程。

当前长期规则：

- 后端启动时自动检查昨日及更早未收口训练课
- 训练相关入口保留兜底收口
- 一组都没做：`absent`
- 做了部分但未完成：`partial_complete`
- 全部做完但未结束：`completed`

不要把该机制描述为“电脑 24 小时开机后零点定时任务”。

当前本地笔记本场景下，正确表述是：

系统启动时自动跨日收口，训练入口保留兜底收口。

未来如果改成常开服务器，可以再升级为真正定时任务。

## 8. 训练端交互长期规则

训练端必须适合 iPad / 平板横屏、触摸操作、多人轮流录入。

### 训练端主入口

动作列表是训练端主入口。

队员应能直接点击动作进入详情录入。

### 动作详情页

动作详情页应满足：

- 当前组显示明显
- 历史组可快速查看
- 历史组允许修改，便于修正手误
- 多组同屏优先
- 不要要求队员频繁输入复杂文字
- 不要用复杂弹窗打断操作

### 动作完成

一个动作录满后：

- 自动标记动作完成
- 自动返回动作列表
- 提示应轻量
- 不影响下一个队员继续操作

### 整堂课完成

整堂课录完后：

- 自动进入完成状态
- 提示应轻量
- 保留“结束计划”按钮
- 手动结束未完成课程时，应进入 `partial_complete`

---

## 9. 本地草稿和同步规则

本地草稿和同步机制是长期核心能力，不能随意破坏。

### 基本原则

本地优先，后台同步，失败不打断训练。

### 必须满足

- 每堂训练课有本地草稿
- 每录完一组先写本地
- 后台再尝试同步到后端
- 同步失败不影响继续录入
- 同步状态要清楚
- 自动重试为主
- 长期失败进入人工处理
- 可手动触发整课同步
- 教练能看到同步异常
- 队员不需要理解同步细节

### 当前同步状态

至少包括：

- `synced`
- `pending`
- `manual_retry_required`

含义：

- `synced`：已同步
- `pending`：本地有未同步内容，正在等待或重试
- `manual_retry_required`：自动重试失败，需要教练或管理员处理

### 整课覆盖同步

当增量同步不可靠时，允许走整课覆盖同步兜底。

但明显冲突必须记录：

- 本地快照
- 后端快照
- 冲突原因
- 处理结果
- 时间和对象

### 不允许

- 不允许同步失败后丢本地数据
- 不允许队员端弹复杂错误
- 不允许用“刷新页面”作为主要恢复方案
- 不允许直接清空本地草稿而不确认
- 不允许在未写日志时覆盖明显冲突

---

## 10. 第二阶段：训练实时监控端 MVP 规则

第二阶段重点是做训练实时监控端最小可用版本。

监控端的目标是：

让教练在训练过程中一眼看到全队当前训练进度、每个人做到哪里、是否有同步异常、谁需要关注。

### 第二阶段不做什么

- 不要做成复杂数据大屏
- 不要做复杂 AI 分析
- 不要做复杂报表
- 不要做复杂战术系统
- 不要做漂亮但无用的炫酷图表

### 监控端 MVP 应优先显示

至少包括：

- 当天训练日期
- 队伍筛选
- 运动员列表
- 每名运动员当天训练状态
- 当前正在做的动作
- 已完成动作数 / 总动作数
- 已完成组数 / 总组数
- 最新一组重量、次数、RIR
- 同步状态
- 异常提示
- 点击运动员查看详情

### 监控端状态颜色

颜色语义要稳定，不要随意变：

- 未开始：灰色或红色
- 进行中：黄色或蓝色
- 已完成：绿色
- 未完全完成：橙色
- 缺席：灰色
- 同步异常：红色或警告色

### 监控端刷新策略

第一版可以使用简单轮询。

不要为了 MVP 直接上复杂 WebSocket，除非用户明确要求。

推荐策略：

- 默认每 3 到 5 秒刷新一次
- 刷新不能影响训练端录入
- 页面应有刷新状态提示
- 支持手动刷新
- 数据加载失败时保留上次结果并提示

### 监控端权限边界

当前免登录模式下，不做复杂权限。

但结构上要兼容未来：

- 教练看自己队伍
- 管理员看全部队伍

不要把监控端写死成只能看一个固定队伍。

### 监控端交互边界

MVP 中监控端主要是只读。

除非用户明确要求，不要在监控端加入大量编辑能力。

可以考虑的最小操作：

- 查看运动员训练详情
- 查看同步异常
- 触发手动同步
- 跳转到训练记录详情

不要在第一版监控端加入：

- 大规模批量编辑
- 批量删除
- 复杂补录
- 模板修改
- 动作库修改
- 测试数据导入

## 11. 管理端长期规则

管理端用于配置、维护、回看和修正，不是训练现场的主录入入口。

管理端应包括：

- 运动员管理
- 队伍管理
- 动作库管理
- 训练模板管理
- 计划分配
- 训练数据回看
- 测试数据管理
- 备份恢复
- 日志
- 监控端入口

管理端允许比训练端复杂，但仍应保持中文清晰、操作明确。

## 12. 动作库长期规则

动作库默认全局共享。

动作库改动需要谨慎，因为它会影响模板配置和后续训练计划。

动作库应支持：

- 搜索
- 分类筛选
- 标签筛选
- 新建动作
- 编辑动作
- 删除动作
- 结构化标签
- 原始数据追溯

动作库改动要求：

- 重要改动写日志
- 删除动作必须二次确认
- 不要破坏历史训练记录展示
- 不要把动作库做成临时训练记录的一部分
- 不要让动作标签系统影响训练端录入速度

---

## 13. 训练模板长期规则

训练模板是计划分配的基础。

### 模板规则

- 模板可以编辑
- 未开始训练可以使用模板最新内容
- 已发生历史训练不能被模板变化破坏
- 模板动作顺序就是训练顺序
- 自动调重默认关闭
- 不要在未开启自动调重时显示复杂建议逻辑

### 模板改动要求

- 删除模板必须二次确认
- 重要改动写日志
- 不要让模板编辑误伤历史 session
- 不要为了一个小需求重写模板系统

---

## 14. 计划分配长期规则

Assignment 表示候选训练计划，不是正式训练课。

### 当前适合的分配方式

- 日期范围
- 周几循环
- 单日例外
- 同一天多个候选计划

### 训练端行为

- 当天命中的计划列出来
- 队员或教练选择要执行的计划
- 打开计划不代表正式开始
- 第一组提交后才正式进入训练 session

### 不要过度设计

当前不要默认引入复杂年度周期、复杂训练日历、复杂自动排课。

这些可以以后做，但不能影响当前训练主链稳定。

---

## 15. 测试数据模块长期规则

测试系统不是训练端实时录入主链。

它服务于阶段性测试、能力评估、后续分析。

### 测试数据应支持

- 导入 Excel
- 网页手动补录
- 查看历史测试
- 按运动员查看
- 按队伍查看
- 后续支持趋势分析

### 导入规则

- 导入失败必须明确指出行、列、字段、原因
- 新批次导入默认保留历史
- 某批次数据出错时允许删除后重导
- 删除测试批次必须二次确认
- 不要随意清空所有真实数据

### 当前不要做深

- 不要做复杂智能表头识别
- 不要做复杂测试标准化系统
- 不要自动强绑定训练建议
- 不要把测试系统改成训练端依赖项

---

## 16. 数据库和迁移规则

数据库结构演进必须以正式迁移为主。

当前长期口径：

- 已接入 Alembic 基线 migration
- `backend/scripts/migrate_db.py` 是迁移入口
- `schema_sync.py` 仅作为过渡期兜底
- 不再把 `schema_sync.py` 当长期正式迁移主方案

### 改数据库前必须说明

任何新增表、字段、索引、状态值前，必须说明：

- 为什么要加
- 给哪个模块用
- 和旧结构的关系
- 是否需要迁移脚本
- 是否需要备份
- 是否影响旧数据
- 如何验收
- 如何回退或至少如何恢复备份

### 不允许

- 不允许只改 ORM model，不写迁移
- 不允许只靠启动时补表补字段
- 不允许静默改状态枚举
- 不允许破坏已有真实数据
- 不允许用删除数据库来解决迁移问题，除非用户明确允许且已备份

---

## 17. 备份和恢复规则

备份恢复是项目长期安全底线。

### 必须保留

- 每日自动备份
- 迁移前备份
- 危险操作前备份
- 手动备份
- 备份列表
- 恢复入口
- 恢复日志

### 恢复操作

恢复属于危险操作，必须：

- 二次确认
- 显示恢复范围
- 恢复前自动备份当前库
- 恢复后写日志

输出：

- 改动文件

## 40. Windows 环境搜索工具规则

Windows environment note:
`rg.exe` may be blocked by Windows security policy in this environment.
If `rg` fails with `EPERM` / `Access denied` / `permission denied`, do not retry `rg` repeatedly.
Immediately fall back to `git grep`, PowerShell `Get-ChildItem` + `Select-String`, or Python `pathlib` search.
Never let `rg` failure block coding, review, testing, or documentation updates.

当前仓库在 Windows 本地环境运行，项目路径可能位于 `Documents`、OneDrive 或用户目录下。
代码搜索优先使用 `rg`；如果 `rg` 被系统拒绝执行，必须立即降级到以下可用方案：

- `git grep "关键词"`
- `Get-ChildItem -Recurse -File | Select-String -Pattern "关键词"`
- Python `pathlib` 递归读取文本文件

不要把 `rg.exe` 失败误判为前端、后端、数据库或业务代码问题。
