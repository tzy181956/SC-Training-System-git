# AGENTS.md

本文件用于约束 Codex 在本仓库中的默认工作方式。目标不是重复项目说明，而是保证跨电脑、跨会话接手时，工作顺序一致、改动节奏稳定、文档维护最小且必要。

## 1. 项目工作原则

- 先读文档，后看代码。
- 先总结理解，后提出计划。
- 先小步修改，后再考虑扩展。
- 未确认前，不做大范围重构。
- 信息不明确时，明确写“待确认”，不要编造。
- 以仓库当前状态为准，不凭空设计。
- 一次优先只解决一条业务链路，不顺手扩散改动。

## 2. 开始任何代码修改前必须阅读的文件

默认阅读顺序：

1. `CURRENT_STATUS.md`
2. `CODEX_HANDOFF.md`
3. `PROJECT_CONTEXT.md`
4. `DEVELOPMENT_GUIDE.md`
5. `README.md`
6. `NEXT_STEPS.md`
7. `CHANGELOG.md`

说明：

- 当前状态、当前卡点、当前优先级，优先以 `CURRENT_STATUS.md` 为准。
- 如果文档之间有冲突，优先参考：
  1. `CURRENT_STATUS.md`
  2. `CODEX_HANDOFF.md`
  3. `PROJECT_CONTEXT.md`
- 阅读完文档后，再结合当前目录结构和相关代码定位本轮任务范围。

## 3. 默认工作流程

默认按以下顺序工作：

1. 先读取上述说明文件。
2. 用简洁中文总结当前理解。
3. 明确本轮要处理的问题、影响范围和计划。
4. 若任务边界不清晰，先提出最少必要问题；若边界清晰，再开始修改。
5. 修改时坚持小步提交，不顺手重构无关模块。
6. 修改后做最小必要验证。
7. 按“最小必要原则”更新相关文档，并说明哪些文档不需要更新以及原因。

如果一次任务需要改多个文件，先说明将修改哪些文件，以及每个文件为什么需要改。

## 4. 文档更新规则

每次有实际代码改动后：

- 必须更新 `CHANGELOG.md`。
- 只有当项目当前状态、当前卡点、当前优先级发生明显变化时，才更新 `CURRENT_STATUS.md`。
- 只有当下一步任务或优先级变化时，才更新 `NEXT_STEPS.md`。
- 只有当交接重点、项目约束、当前开发背景发生变化时，才更新 `CODEX_HANDOFF.md`。
- 只有当项目定位、技术结构、启动方式发生明显变化时，才更新 `README.md` 或 `PROJECT_CONTEXT.md`。
- `DEVELOPMENT_GUIDE.md` 只在开发约束、分层规则、验证方式发生变化时更新。

不要机械重写所有文档。
如果某个文档本轮不需要更新，要明确说明“不更新的原因”。

## 5. 修改代码时的约束

- 不要顺手改无关模块。
- 不要轻易改架构。
- 优先保持已有稳定功能可运行。
- 优先保证可验证、可继续迭代，而不是一次性做大而全。
- 训练模板和动作库默认视为保留资产，不要为了清理数据顺手重置。
- 训练模式主流程除非明确修 bug，不要顺手重构。
- Windows 一键启动和初始化脚本改动前，要先确认现有行为。
- 真实数据导入链路优先增强和修正，不要随意推翻。
- 优先保持以下链路稳定：
  - `/training`
  - `/assignments`
  - 真实数据导入与测试记录展示
  - 运动员主档身体指标字段

## 6. 输出与协作方式

- 默认使用简洁中文说明。
- 先给“当前理解总结”，再给“本轮计划”。
- 对无法从仓库确认的信息，直接标记“待确认”。
- 没有得到明确任务前，不主动做无关重构或额外优化。
