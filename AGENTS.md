你现在不是在“顺手修 bug”或“顺手加功能”。

你是这个项目的**重构工程师**。

## 6. 关键业务术语（统一使用）

为了避免术语混乱，优先使用下面这些定义。

### Template
训练模板。  
表示一份可复用的训练内容原型。

### Assignment
计划分配 / 候选训练计划。  
不是一堂课本身。  
它表示：某份模板在某个时间范围内，对某个运动员可被命中并用于训练。

### Session
一堂训练课的正式记录。  
不是模板，不是 assignment。  
它是某一天某次真实训练的正式载体。

### Local Draft
训练端本地草稿。  
当前实现是浏览器本地 `localStorage` 中的 session 离线镜像，用于同一浏览器 / 同一设备内的本地优先保存与同步兜底，不等于跨浏览器、跨设备或清缓存后仍可恢复的强持久化方案。

### Set Record
某个动作某一组的实际训练记录，例如重量、次数、RIR 等。

### Sync Status
同步状态。  
用于标记本地草稿 / session 是否已经成功同步，是否异常待处理。

## 7. 训练课状态流（必须严格遵守）

第一阶段中，训练课状态至少包括：

- `not_started`：未开始
- `in_progress`：进行中
- `completed`：完成
- `absent`：缺席
- `partial_complete`：未完全按计划完成

### 状态进入规则

#### not_started
- assignment 已命中
- 但还没有正式开始训练

#### in_progress
- 第一组训练数据提交成功后进入
- 这时应正式创建/确认 session

#### completed
- 所有应做内容完成
- 或系统按规则自动完成整堂课

#### absent
- 触发跨日收口时，一组都没做

#### partial_complete
- 已做了部分内容
- 但没有按计划完成整堂课

### 状态边界规则

1. 打开计划不算开始
2. 查看内容不算开始
3. 第一组提交成功，才算正式开始
4. 一组都没做，触发跨日收口时记为 `absent`
5. 做了部分但没做完，触发跨日收口时记为 `partial_complete`
6. 训练结束后，教练允许课后修正，但要写日志
7. 课后补录组数据后，应自动重算状态

## 8. 训练端规则（必须写死）

### 总原则
训练端稳定优先于一切。

### 训练端必须满足
1. 队员录入不能被网络问题打断
2. 每录完一组，先写入本地草稿
3. 队员不需要理解同步状态
4. 队员不负责处理同步失败
5. 教练课后可以看到同步异常
6. 训练端 UI 结构尽量沿用现有，不做大改

### 训练端交互规则
1. 动作列表是主入口
2. 队员自己点某个动作进入录入
3. 动作详情页支持多组同屏
4. 当前组必须更突出
5. 历史组允许快速修改，便于修正手误
6. 一个动作录完后自动标记完成并回动作列表
7. 整堂课录完后自动结束，但提示必须很轻，不打断下一个人
8. 保留“结束计划”按钮
9. 跨日未收口训练课会在后端下次启动时自动收口，训练入口保留兜底收口

### 当前阶段先不做深的现场变化
以下现场变化当前阶段先不做深：
- 临时删动作但不记录原因
- 临时改动作顺序
- 临时把一个动作换成另一个动作

以后如果补，优先补：
- 跳过动作按钮

## 9. 本地草稿与同步规则（必须写死）

### 总原则
本地优先，后台同步。  
以训练端连续录入体验为第一优先级。

当前实现的本地草稿以浏览器 `localStorage` 为主，主要覆盖同一浏览器 / 同一设备内的断网续录、页面异常关闭恢复与课后补传；不要把它描述成跨浏览器、跨设备或清缓存后仍然可靠的强持久化能力。

### 基本规则
1. 每堂训练课有本地草稿
2. 本地草稿是 session 的离线镜像
3. 第一组提交成功时，正式创建/确认 session
4. 每录完一组先写本地
5. 后台优先尝试增量补传
6. 如果增量补传不稳，允许整堂课覆盖同步兜底
7. 大多数情况下以本地草稿最终结果为准覆盖后端
8. 明显冲突时必须提示教练端/管理员端
9. 自动重试为主
10. 长时间失败后转入“同步异常，待处理”
11. 允许人工手动触发重试

### 同步状态至少要统一
至少统一这些同步状态：
- 已同步
- 未同步
- 同步中
- 同步异常待处理

### 同步状态可见范围
第一阶段要有基础版可见性：
- 训练端设备本身
- 教练 / 监控端
- 管理端 / 日志页

### 同步失败处理底线
1. 不打断训练
2. 尽量保住当前浏览器本机草稿数据
3. 冲突要有日志
4. 另一台设备短时间内尽量能看到更新
5. 同步异常要能被教练或管理员处理

## 12. 测试系统规则（第一阶段边界）

测试系统不是当前阶段主战场，只维持可用和可导。

### 当前原则
1. 测试结构保持相对自由
2. 网页导入和脚本导入都允许
3. 网页手动添加主要服务于补录个别运动员/个别指标
4. 正常新批次导入时保留历史、新增本次数据
5. 某一批数据出错时允许删除后重导
6. 导入失败必须明确指出：
   - 哪一行
   - 哪一列
   - 哪个字段
   - 是格式错误还是数据内容错误
7. 当前不要深做智能表头识别、复杂标准化、复杂训练建议绑定

## 13. 启动器 / 初始化 / 迁移 / 备份 / 恢复

### 启动器目标
让非程序员也大致知道系统为什么启动失败、该怎么修。

### 启动器必须做到
1. 一键初始化 + 一键启动保留
2. 新电脑首次启动时，一步步检查、一步步初始化、最后启动
3. 失败时尽量自动修复一部分
4. 修不了时明确告诉用户：
   - 哪一步失败
   - 可能原因
   - 修复建议
5. 最好输出可复制错误摘要，便于发给 AI 或别人排查

### 正式迁移机制
1. 不再长期依赖“启动时补字段/修表”
2. 数据库结构变更必须走正式迁移机制
3. 迁移前必须自动备份
4. 迁移前要明确提示迁移内容和风险

### 自动备份
第一版至少支持：
1. 每日定时自动备份
2. 危险操作前自动备份
3. 迁移前自动备份
4. 保留最近几天 + 最近几周的备份

### 恢复
第一版希望逐步支持：
1. 整库恢复
2. 某类数据恢复
3. 恢复后明确提示恢复到了哪个时间点
4. 恢复操作必须写日志

## 14. 危险操作规则（必须写死）

以下操作属于危险操作：
- 删除模板
- 删除训练记录
- 删除某批测试数据
- 清空后重导
- 恢复备份
- 触发可能覆盖数据的同步修复

### 危险操作必须满足
1. 二次确认
2. 明确影响范围
3. 建议先备份
4. 写操作日志

## 15. 日志规则（第一版基础版就要有）

第一版日志至少覆盖：

1. 训练记录被修改 / 补录
2. 模板被修改
3. 动作库被修改
4. 删除、清空重导、恢复备份等危险操作
5. 同步失败
6. 同步冲突

### 日志页基础能力
第一版最好支持基础筛选：
- 时间
- 操作人

### 日志明细最低要求
- 时间
- 谁做的
- 做了什么
- 影响了哪个对象
- 改前/改后关键字段（能做则做）

## 17. 多平板一致性规则

### 当前原则
1. 训练端本地录入那台设备优先最稳
2. 另一台设备尽量短时间内看到更新
3. 冲突出现时必须有清晰提示
4. 教练端和管理员端都能处理冲突
5. 不要为了实时一致牺牲训练端稳定

### 当前接受的实现方向
1. 一边录入，另一边尽量自动刷新看到结果
2. 自动刷新 / 推送都可，只要不伤训练端
3. 若同一组被不同设备同时改：
   - 本地当前修改优先
   - 记录冲突
   - 提示教练端/管理员端处理

---

## 18. 当前阶段代码改造原则

1. 为了清晰允许明显重组目录结构
2. 能删掉的旧逻辑尽量删，不保留两套长期并存的核心逻辑
3. 前端页面结构尽量沿用现有，不做大改 UI
4. 前端只改训练端必要逻辑
5. 后端优先修状态流和同步主链，不顺手扩别的系统
6. 新增表/字段时，必须写清：
   - 为什么加
   - 给谁用
   - 和旧结构关系
   - 迁移顺序
   - 回退注意事项
7. 至少补足必要注释和文档字符串
8. 统一接口命名、统一字段、统一状态枚举，避免前端到处特判
